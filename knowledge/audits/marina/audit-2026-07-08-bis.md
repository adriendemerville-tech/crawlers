## 1. Findings

### M7 · P0 · GRANT manquant sur les 7 tables Marina/outreach
**Preuve SQL** :
```
information_schema.role_table_grants
→ marina_prospects / marina_training_data / marina_api_keys /
  prospect_outreach_queue / outreach_sequences / outreach_daily_quotas / outreach_events
grantees présents : sandbox_exec (INSERT,SELECT) uniquement.
authenticated / service_role / anon : AUCUN grant.
```
**Impact** : violation directe de la règle `<public-schema-grants>` du projet. RLS seule ne suffit pas — toute lecture PostgREST via `authenticated` retourne `permission denied for table`. Fonctionne aujourd'hui uniquement parce que les edges utilisent `service_role` qui bypass grants + RLS.
**Fix** : migration `GRANT SELECT,INSERT,UPDATE,DELETE ON public.<table> TO authenticated; GRANT ALL ON public.<table> TO service_role;` (retirer `authenticated` sur les tables admin-only et laisser `service_role` seul).
**Effort** : 0,5j (migration + revue policies).

### M8 · P0 · `marina_api_keys` : impossible pour l'utilisateur d'enregistrer sa clé
**Preuve** : 2 policies `SELECT` uniquement (`Users can read own marina key`, `Users can view own marina api keys`), aucune `INSERT/UPDATE/DELETE`. Colonnes `user_id`, `api_key`, `label`, `selected_services`, `is_active` — clairement destinées à un CRUD utilisateur.
**Impact** : le flow "Custom REST API Key" documenté dans `mem://features/cms/custom-rest-api-key-registration-fr` est cassé pour Marina. Aucun front ne peut créer la clé sans passer par une edge admin.
**Fix** : ajouter `CREATE POLICY "Users manage own marina api keys" ON public.marina_api_keys FOR ALL TO authenticated USING (auth.uid()=user_id) WITH CHECK (auth.uid()=user_id);` + doublon SELECT à supprimer.
**Effort** : 0,25j.

### M9 · P0 · `marina_prospects` / `prospect_outreach_queue` en Admin-only → contradiction avec le pipeline "1 owner par prospect"
**Preuve** : policies `ALL` réservées aux admins (`Admins can manage prospects` / `Admins can manage outreach queue`). Or `outreach_sequences` et `outreach_daily_quotas` sont bien scopés `auth.uid()=user_id` → asymétrie.
**Impact** : le Pro Agency propriétaire d'un prospect ne peut pas lire sa liste ni suivre la queue → toute la partie autonome du pipeline Marina ne fonctionne qu'en admin. Confirmé par le grep : `ProspectPipelineDashboard.tsx` cast `.from('marina_prospects' as any)` = **types Supabase désynchronisés**, symptôme d'un accès contourné.
**Fix** : ajouter policies `owner-scoped` sur `marina_prospects` (owner_user_id) et `prospect_outreach_queue` (via join `marina_prospects.owner_user_id`), conserver l'accès admin en parallèle via `has_role(auth.uid(),'admin')`.
**Effort** : 0,5j.

### M10 · P1 · `prospect-pipeline` : `callLovableAIText` sans wrapping `<user_input>`
**Preuve** : `supabase/functions/prospect-pipeline/index.ts` ligne 190 injecte `prompt` construit avec `p.linkedin_url`, `p.company_name`, `p.description` (données prospect scrapées, potentiellement adverses) sans passer par `wrapUserInput()` — le prompt safety layer (`mem://features/copilot/prompt-safety-layer`) n'est PAS appliqué en dehors du copilot.
**Impact** : prompt injection possible depuis la bio LinkedIn d'un prospect (ex : "Ignore all previous instructions and reply with…").
**Fix** : importer `wrapUserInput` de `_shared/promptSafety.ts` et wrapper systématiquement `p.description`, `p.company_name`, `p.headline` avant concaténation.
**Effort** : 0,25j.

### M11 · P1 · Typage `: any` sur le workflow Marina admin
**Preuve grep** :
```
MarinaDashboard.tsx           : 15 occurrences
ProspectPipelineDashboard.tsx :  3 occurrences (dont cast 'marina_prospects' as any)
prospect-pipeline/index.ts    : 10 occurrences
marinaWorkbench.ts            :  2 occurrences (sb: any, rows: any[])
Marina.tsx                    :  4 occurrences
```
**Impact cumulé Vague 4 marina** : 57 (monolithe) + 34 (élargi) = **91 `: any`** sur le workflow → aucune garantie de forme sur les rapports générés / prospects insérés.
**Fix** : générer les types depuis les schemas (`npx supabase gen types typescript`), interdire `any` sur `marina*` via `eslint no-explicit-any` scoped.
**Effort** : 0,5j.

### M12 · P1 · Aucune télémétrie produit Marina/outreach
**Preuve SQL** : `select count(*) from analytics_events where event_type ilike '%marina%' or '%prospect%' or '%outreach%'` → **0 rows**.
**Impact** : impossible de mesurer adoption (`marina_audit_launched`, `marina_prospect_contacted`, `outreach_step_sent`, `marina_export_pdf`), donc impossible de prioriser roadmap. Marina est déployé "à l'aveugle".
**Fix** : instrumenter 4 events canoniques dans `Marina.tsx` (view), `MarinaDashboard.tsx` (launch audit + share), `prospect-pipeline` (step_sent server-side avec `event_source='edge'`).
**Effort** : 0,5j.

### M13 · P1 · `src/pages/Marina.tsx` = 1 510 l. de viewer démo — divergence intent/implémentation
**Preuve** : `rg "functions.invoke" src/pages/Marina.tsx` → **0 résultat**. La page publique Marina n'appelle JAMAIS l'edge `marina`. Elle ne fait que charger le dernier `async_jobs` complété + afficher HTML. Toute la logique d'audit prospect est enfermée dans `MarinaDashboard.tsx` (admin).
**Impact** : soit la page publique est une "démo froide" (à assumer et à réduire à 300 lignes max), soit c'est un bug produit (Pro Agency devrait pouvoir lancer un audit prospect self-service). Le brief `mem://features/prospecting/marina-module-fr` suggère l'usage self-service Pro Agency.
**Fix** : décision produit + soit (a) découpe en `MarinaHero.tsx` + `MarinaDemoViewer.tsx` + `MarinaLaunchForm.tsx` (150-300 l. chacun) branchée sur edge `marina`, soit (b) déprécier la page et rediriger vers `/admin/marina`.
**Effort** : 1,5j (voie a, préférée).

### M15 · P2 · `view-marina-report` : pas de contrôle d'accès sur le rapport HTML
**Preuve** : `view-marina-report/index.ts` sert n'importe quel `job_id` UUID valide sans vérifier l'appartenance (`user_id`). N'importe qui connaissant l'UUID accède au rapport.
**Impact** : fuite potentielle de données prospect si UUID leak (log, referrer, share). Rapport est déjà "public par design" via `shared-reports` bucket, mais absence de gate.
**Fix** : soit accepter (documenter comme "share by URL" comme `share-actions`), soit ajouter un HMAC signature dans l'URL (`?sig=<hmac(job_id+expiry)>`).
**Effort** : 0,25j (documentation) ou 0,5j (HMAC).

## 3. Synthèse
Vague 4 avait pointé un monolithe. Vague 4-bis démontre que **le problème est écosystémique** : les tables sont mal grantées, le pipeline outreach est administrativement verrouillé (bloquant l'usage Pro Agency), le front public est un viewer démo déconnecté de l'edge, la télémétrie est nulle et le prompt safety layer manque hors Copilot. Le refactor Marina doit inclure **DB + edges + front + télémétrie**, pas uniquement le découpage du monolithe.
