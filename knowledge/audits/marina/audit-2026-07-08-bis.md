# Audit — SERP Benchmark multi-providers
**Date :** 2026-07-08 · **Auditeur :** Lovable · **Feature :** `serp-benchmark`

---

## Résumé

Le cœur métier (4 providers en parallèle, pondération, single-hit penalty, drift workbench) est **correctement implémenté** et constitue un différenciateur réel vs Semrush/Ahrefs. Cependant, **4 défauts bloquants** sabotent la promesse commerciale et la sécurité : variables `supabaseUrl`/`serviceKey` non déclarées dans `refresh-serp-all` (crash silencieux du cron), zéro rate-limit sur le lead magnet anonyme (drain coût), zéro tracking coût dans `serp-benchmark` (invisible dans `paid_api_calls`), et la table `serpapi_cache` jamais lue par `serp-benchmark` (0% cache hit, gaspillage 100%). **Score global : 4.5/10.**

---

## P0 — Critiques (production cassée / drain immédiat)

### P0-1 · `supabaseUrl` et `serviceKey` non déclarés dans `refresh-serp-all`
| | |
|---|---|
| **Fichier** | `refresh-serp-all/index.ts:259, 262, 439, 442, 501, 504` |
| **Preuve** | `fetch(\`${supabaseUrl}/functions/v1/fetch-serp-kpis\`, { headers: { Authorization: \`Bearer ${serviceKey}\` } })` — aucun `import` ni `Deno.env.get` pour ces deux variables dans tout le fichier |
| **Impact** | `ReferenceError: supabaseUrl is not defined` à chaque appel de la boucle SERP (ligne 259) → la boucle de batch saute le step SERP pour **tous** les sites. Idem pour Cocoon (439) et la self-invocation (501). Le cron tourne mais ne fait rien d'utile. |
| **Fix** | Ajouter en tête du handler : `const supabaseUrl = Deno.env.get('SUPABASE_URL')!;` et `const serviceKey = Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!;` |
| **Effort** | 15 min |
| **Score** | 1/10 |

### P0-2 · Zéro rate-limit sur `action='benchmark'` anonyme
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts:254-299` |
| **Preuve** | Le handler accepte `action='benchmark'` sans auth (`user` peut être `null` l.304) avec aucun appel à `ipRateLimiter`, `verify-turnstile`, ni token bucket. `providers = ['DataForSEO', 'SerpApi', 'Serper']` par défaut. |
| **Impact** | Un script abuse → 3 appels providers payants par requête, sans limite. DataForSEO ~$0.002/call SERP live, SerpApi $0.01-0.05/call. 10 000 requêtes = ~$150-500 de drain, invisible (P0-3). |
| **Fix** | Ajouter `await ipRateLimiter(req, { maxPerMinute: 5, keyPrefix: 'serp-anon' })` pour les requêtes non-authentifiées ; ou exiger Turnstile token sur lead magnet. |
| **Effort** | 2h |
| **Score** | 1/10 |

## P1 — Hauts (fonctionnel dégradé / coût invisible)

### P1-1 · Zéro `trackPaidApiCall` dans `serp-benchmark`
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts` (446 lignes, 0 occurrence) |
| **Preuve** | `grep trackPaidApiCall serp-benchmark/index.ts` → vide. Contraste : `fetch-serp-kpis:166,245,383` et `refresh-serp-all:168,186,383` trackent proprement. |
| **Impact** | 3 appels providers par benchmark = coût 100% invisible dans `paid_api_calls`. Baseline brief confirme 0 rows attendues. Impossible de calculer le coût par benchmark, de facturer les Agency, ou de déclencher `dailyCostGuard`. |
| **Fix** | Après chaque fetcher réussi, appeler `trackPaidApiCall('serp-benchmark', provider.toLowerCase(), endpoint)`. |
| **Effort** | 1h |
| **Score** | 3/10 |

### P1-2 · `serpapi_cache` jamais lue par `serp-benchmark` — cache hit 0%
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts` (0 référence à `serpapi_cache`) vs `serpapi-actions/index.ts:83-103` (cache lu correctement) |
| **Preuve** | `grep serpapi_cache serp-benchmark/index.ts` → vide. Brief confirme baseline : 0 rows en cache venant de `serp-benchmark`. |
| **Impact** | Queries identiques (ex: batch top keywords Pro Agency) appellent 3× les providers même si la même query a été faite < 1h. Gaspillage 100% sur les benchmarks récurrents. La table `serpapi_cache` (16 col, 4 policies) existe mais est morte pour ce use case. |
| **Fix** | Avant les `fetchers.push`, vérifier `serp_benchmark_results` (ou `serpapi_cache`) pour une query identique < 24h. Retourner `averaged_results` du cache si trouvé. |
| **Effort** | 3h |
| **Score** | 3/10 |

### P1-3 · `Promise.all` fragile — devrait être `Promise.allSettled`
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts:299` |
| **Preuve** | `const providerResults = await Promise.all(fetchers)` — les fetchers catchent leurs erreurs aujourd'hui, mais `fetchBrightData` effectue `new URL(...)` non protégé sur réponse arbitraire (l.152-153). Un `throw` non attrapé = benchmark entier rejeté. |
| **Fix** | Remplacer par `Promise.allSettled` + mapper les `fulfilled`/`rejected` : `fetchers.map(async f => { try { return await f } catch(e) { return { provider: '?', results: [], error: String(e) } } })` |
| **Effort** | 30 min |
| **Score** | 5/10 |

### P1-4 · Doublon `fetchSerpApi` inline vs `serpapi-actions`
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts:61-91` vs `serpapi-actions/index.ts:56-181` |
| **Preuve** | Deux implémentations indépendantes du même endpoint SerpApi. `serpapi-actions` a : cache 24h, analytics_events tracking, gestion multi-engine (local, news, images). `serp-benchmark` a : rien de tout ça. |
| **Fix** | Extraire dans `_shared/serpProviders/serpApi.ts` une fonction `fetchSerpApiRaw(query, opts)` partagée. `serpapi-actions` et `serp-benchmark` l'importent. |
| **Effort** | 4h |
| **Score** | 4/10 |

## P2 — Moyens (qualité / maintenabilité)

### P2-1 · Prompt LLM non wrappé avec données user dans `fetch-serp-kpis`
| | |
|---|---|
| **Fichier** | `fetch-serp-kpis/index.ts:259-288` |
| **Preuve** | `identityDesc` = concaténation directe de `site_context.products_services`, `.market_sector`, `.target_audience`, `.commercial_area` (champs user) injectée dans `content` système LLM sans `promptSafety` wrapper. `_shared/promptSafety.ts` existe et exige `<user_input>…</user_input>`. |
| **Fix** | Wrapper `identityDesc` avec `wrapUserContent(identityDesc)` (ou balises manuelles) ; passer les données contrôlées uniquement dans le rôle `user`, pas `system`. |
| **Effort** | 1h |
| **Score** | 5/10 |

### P2-2 · `: any` prolifération
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts` : 9 × `: any` · `fetch-serp-kpis/index.ts` : 11 × `: any` |
| **Preuve** | `grep -c ': any'` → 9 et 11. Exemples : `(i: any)` l.48-49, `catch (e: any)` × 4, `(r: any)` × 5 dans `fetch-serp-kpis`. |
| **Fix** | Typer les réponses providers (DataForSEO, SerpApi, Serper) avec des interfaces explicites ; utiliser `unknown` + type guard dans les `catch`. |
| **Effort** | 3h |
| **Score** | 5/10 |

### P2-3 · UX : emojis et couleurs bleues IA interdits
| | |
|---|---|
| **Fichier** | `SerpBenchmark.tsx:34` (provider colors) · `SerpBenchmark.tsx:409` (emoji ❌) |
| **Preuve** | `{ id: 'DataForSEO', color: 'bg-blue-500' }` — brief §4 : "pas d'emoji, pas de bleu IA". Badge DataForSEO = bleu IA direct. Ligne 409 : `❌ ${p.error}`. |
| **Fix** | Remplacer `bg-blue-500` → `bg-slate-700` pour DataForSEO ; supprimer `❌`, utiliser icône Lucide `XCircle` ou texte `Erreur:`. |
| **Effort** | 30 min |
| **Score** | 6/10 |

### P2-4 · `computeAveragedResults` — fallback hostname `"unknown"` silencieux
| | |
|---|---|
| **Fichier** | `serp-benchmark/index.ts:195` |
| **Preuve** | `new URL(r.url \|\| 'https://unknown').hostname` → retourne la string `"unknown"` si `r.url` est vide/invalide, qui pollue silencieusement le groupement de domaines dans `urlMap`. |
| **Fix** | Tester `r.url` avec `URL.canParse()` avant ; sinon `skip` (ne pas ajouter à `urlMap`) plutôt que de créer un domaine fantôme. |
| **Effort** | 30 min |
| **Score** | 6/10 |

### P2-5 · `refresh-serp-all` — scope creep (7 responsabilités, 517 lignes)
| | |
|---|---|
| **Fichier** | `refresh-serp-all/index.ts` |
| **Preuve** | Une seule fonction gère : SERP (DataForSEO), GSC weekly, backlinks (DataForSEO), GA4, `user_stats_history` compat, Cocoon refresh (Pro Agency), nettoyage cocoon_chat_histories/sessions. |
| **Fix** | Extraire le cleanup (§7) dans `cron-cleanup` séparé ; Cocoon refresh (§6) dans `cron-cocoon-refresh` (déjà existant — doublon potentiel). |
| **Effort** | 1 jour |
| **Score** | 5/10 |
