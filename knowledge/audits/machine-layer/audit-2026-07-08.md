## #286 · P2 · 1j — SSRF : DNS rebinding non couvert par `assertSafeUrl`

`assertSafeUrl` filtre les IPs privées par regex sur le hostname *avant* résolution DNS. Un attaquant peut enregistrer un domaine public dont le TTL expire en mid-flight (DNS rebinding) ou utiliser un redirecteur `302 → http://169.254.169.254/` : le check passe sur l'URL initiale, le redirect atterrit sur l'IP interne. De plus `stealthFetch` suit les redirects sans rappeler `assertSafeUrl` sur l'URL finale.
**Fix ①** Résoudre le hostname via `Deno.resolveDns` avant chaque fetch et rejeter si l'IP résolue est privée.
**Fix ②** Désactiver `redirect: 'follow'` sur `stealthFetch`, gérer les redirects manuellement en appelant `assertSafeUrl` à chaque hop (max 3).

## #287 · P2 · 0.5j — Timeout cascade SPA : 165 s cumulables sans cap global

`TIMEOUT_MS = 15_000` ne couvre que le fetch initial. La cascade SPA enchaîne Spider (`AbortSignal.timeout(30000)`) + Browserless (`AbortSignal.timeout(60000)`) + Fly (`AbortSignal.timeout(60000)`) sans signal partagé, soit jusqu'à 165 s de wall-time par requête. Une SPA + Spider KO bloque le worker Deno pendant ~90 s avant même Browserless.
**Fix ①** Introduire un `AbortSignal.timeout(TOTAL_TIMEOUT_MS = 45_000)` passé à chaque étape de la cascade.
**Fix ②** Mettre la LLM call (`AbortSignal.timeout(45000)`) sous le même signal partagé ou réduire à 20 s.

## #288 · P2 · 1j — Coût LLM : tokens non plafonnés, aucun budget guard par scan

`buildRecommendations` injecte jusqu'à 25 issues + l'intégralité du contexte signaux dans le prompt sans `max_tokens`. Sur un site avec 25 issues critiques, le prompt dépasse 4 k tokens ; la réponse peut atteindre 8 k tokens (HTML prêt-à-coller pour chaque issue). Avec `google/gemini-3-flash-preview`, ~$0.003/scan semble faible mais le lead magnet gratuit (pas de plan check) le rend scalable à l'infini.
**Fix ①** Ajouter `max_tokens: 2048` dans le body de l'appel `aiGatewayFetch`.
**Fix ②** Limiter `scoring.issues.slice(0, 10)` pour le prompt LLM et compléter avec `deterministicSnippet` pour les issues 11-25.

## #289 · P1 · 2j — Gating plan absent : scan LLM gratuit, quota illimité

Aucun contrôle de plan dans l'edge function. Un utilisateur free (ou anonyme avec Turnstile) déclenche la même pipeline que Pro : fetch cascade + LLM + persistance. La landing page affiche explicitement « Sans inscription » et les deux CTAs renvoient vers `/app/machine-layer`. Il n'existe aucune table de quota, aucun compteur `scans_used / scans_limit` vérifié côté edge.
**Fix ①** Vérifier le plan Stripe de l'utilisateur connecté (ou incrémenter un compteur anonyme en KV/Redis) avant l'appel LLM ; renvoyer `429` avec `upgrade_url` si quota dépassé.
**Fix ②** Réserver la génération LLM (`buildRecommendations` avec `aiGatewayFetch`) aux plans payants ; les anonymes et free reçoivent uniquement le fallback déterministe.

## #290 · P2 · 0.5j — RLS SELECT : scans anonymes exposés à tous les utilisateurs authentifiés

La policy `"Users see own scans, anon scans public, admin sees all"` comporte la clause `user_id IS NULL` sans restriction d'IP ni de session. Tout utilisateur authentifié peut lire l'ensemble des scans anonymes (URL scannée, score, signaux, recommandations, `ip_hash`). Les données `detected_signals` incluent parfois des métadonnées sensibles (author, generator, API endpoints dans JSON-LD). `useMachineLayerStatus` côté hook filtre sur `user_id` si connecté, mais n'empêche pas une requête directe à l'API Supabase.
**Fix ①** Supprimer la clause `user_id IS NULL` de la policy SELECT ou la restreindre à `anon` role uniquement (via `auth.role() = 'anon'`).
**Fix ②** Dans `useMachineLayerStatus`, ne jamais requêter les scans sans `user_id` filter quand `user` est non-null.

## #291 · P1 · 1j — XSS potentiel : `ready_to_paste` LLM non sanitisé stocké et renvoyé

Le champ `ready_to_paste` est généré par le LLM (Gemini) et contient du HTML brut (`<script type="application/ld+json">…</script>`, balises meta, etc.). Il est stocké tel quel dans `recommendations JSONB` via service role (bypass RLS), puis retourné dans la réponse JSON. Si le frontend consommateur fait `dangerouslySetInnerHTML` ou `innerHTML` sur ce champ (pattern courant pour un éditeur "prêt à coller"), un LLM compromis ou un scan poisonné peut injecter du JS arbitraire.
**Fix ①** Côté edge, passer `ready_to_paste` dans un sanitizer (DOMPurify-compatible en Deno ou regex stricte de tag allowlist) avant insertion.
**Fix ②** Côté frontend (page `/app/machine-layer`, hors périmètre audité), utiliser exclusivement `textContent` ou un composant `<CodeBlock>` et non `innerHTML` pour afficher ce champ.

## #292 · P3 · 1j — Absence de cache et de déduplication des scans récents

Chaque appel POST déclenche un fetch complet + LLM, même si la même URL a été scannée il y a 2 minutes. Le hook `useMachineLayerStatus` expose un statut `fresh` (< 24 h) mais l'edge function n'interroge pas la table avant de relancer le pipeline. Il n'existe pas de cache HTTP (pas d'en-tête `Cache-Control` en réponse) ni de memoïzation in-process.
**Fix ①** En début de handler, vérifier si un scan `created_at > now() - interval '1 hour'` pour la même URL+user_id existe (via `adminClient`). Si oui, retourner le scan existant sans recalcul.
**Fix ②** En réponse JSON, ajouter `Cache-Control: private, max-age=3600` et un `ETag` basé sur `scan.id` pour permettre le cache navigateur/CDN.

## #293 · P2 · 0.5j — Doublons user_id : absence de contrainte UNIQUE (user_id, url)

La migration ne pose aucune contrainte `UNIQUE` sur `(user_id, url)` ni sur `(ip_hash, url, created_at::date)`. Un utilisateur peut scanner la même URL 10 000 fois (scripting ou boucle frontend) et générer 10 000 lignes + 10 000 appels LLM. L'index `idx_machine_layer_scans_user` n'est pas `UNIQUE`. Aucune protection côté edge.
**Fix ①** Ajouter une contrainte (ou index partiel) : `CREATE UNIQUE INDEX ON machine_layer_scans (user_id, url) WHERE user_id IS NOT NULL;` et une contrainte similaire sur `(ip_hash, url, (created_at::date))` pour les anonymes.
**Fix ②** Côté edge, tester un `ON CONFLICT (user_id, url) DO UPDATE SET updated_at = now(), score_global = EXCLUDED.score_global …` plutôt qu'un INSERT brut.

## #294 · P1 · 1j — Turnstile fail-open + bypass littéral `TURNSTILE_UNAVAILABLE`

Deux failles combinées : (1) ligne 711, le token `'TURNSTILE_UNAVAILABLE'` court-circuite entièrement la vérification Turnstile sans aucune contre-mesure alternative (rate limit IP, ban temporaire). (2) ligne 724, tout `fetch` en erreur vers `challenges.cloudflare.com` est silencieusement ignoré (`/* fail open */`), rendant le bot-check inopérant en cas de panne réseau ou de timeout Cloudflare.
**Fix ①** Supprimer le bypass `TURNSTILE_UNAVAILABLE` ou le remplacer par un secret en env (`TURNSTILE_BYPASS_SECRET`) jamais exposé au client.
**Fix ②** Transformer le `catch` du Turnstile en fail-closed : renvoyer `jsonError('Vérification anti-bot indisponible', 503)` au lieu de continuer silencieusement.

## #295 · P3 · 0.5j — Design System : `noindex` sur page marketing + CTAs non-primaires + `any` cast

Trois problèmes de cohérence : (1) `<meta name="robots" content="noindex,follow" />` sur `MachineLayerLanding.tsx` — une page marketing pilier de la stratégie GEO est volontairement exclue de l'index Google, contredisant l'objectif de visibilité. (2) Les deux CTAs utilisent `variant="outline"` alors que les pages publiques Crawlers utilisent le `variant` primaire (violet) pour l'action principale. (3) Dans l'edge function (lignes 786-787), `signals as any` et `recommendations as any` contournent le typage TypeScript fort sur l'insert Supabase, masquant tout drift de schéma.
**Fix ①** Remplacer `noindex,follow` par `index,follow` sur la landing, ou supprimer la balise pour hériter du défaut.
**Fix ②** Typer explicitement les colonnes JSONB via le type généré Supabase (`Database['public']['Tables']['machine_layer_scans']['Insert']`) et passer le premier CTA en `variant="default"`.
