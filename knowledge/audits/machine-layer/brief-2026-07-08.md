# Audit Brief : Machine Layer (SEO/GEO Scanning)
**Date :** 2026-07-08
**Auditeur :** Senior Crawlers Auditor
**Statut :** Initialisation

## 1. Objectif & Contexte
L'initiative Machine Layer constitue le pivot de la stratégie de visibilité IA de Crawlers. L'objectif est de segmenter et d'optimiser la perception des actifs numériques non plus seulement pour les humains ou les robots d'indexation classiques, mais spécifiquement pour la "couche machine" (LLMs, Search Generative Experience, assistants IA). Ce scan permet de mesurer la lisibilité structurelle des données par les agents autonomes.

## 2. Périmètre Technique
Le périmètre d'audit englobe les composants critiques suivants :
- **Edge Functions :** `machine-layer-scan` et l'ensemble des fonctions préfixées `machine-layer-*`.
- **Persistance :** Table PostgreSQL `machine_layer_scans` (historique et résultats bruts).
- **Interface Utilisateur :**
    - `src/pages/MachineLayerLanding.tsx` (Point d'entrée principal).
    - `src/hooks/useMachineLayerStatus.ts` (Gestion d'état et synchronisation temps réel).
- **Logique Métier :** Analyse des métadonnées, schémas JSON-LD et signaux GEO-SEO.

## 3. Flux & Architectures
Le flux de données suit une séquence asynchrone :
1. Déclenchement du scan via l'interface utilisateur.
2. Appel à l'Edge Function `machine-layer-scan`.
3. Extraction des vecteurs de visibilité machine (Metadata, Schema.org, Accessibilité sémantique).
4. Stockage des résultats dans la table `machine_layer_scans`.
5. Mise à jour de l'UI via le hook de statut dédié.

## 4. Performance & Métriques
Les indicateurs clés de performance (KPI) pour cet audit incluent :
- Temps de réponse moyen de l'Edge Function (latence froide/chaude).
- Taux de complétion des scans (succès vs échecs de parsing).
- Profondeur d'analyse des balises sémantiques.
- Précision du scoring de "Machine Readability".

## 6. Risques & Points de Vigilance
- **Divergence de Perception :** Écart entre le rendu DOM pour un humain et le code source brut analysé par les bots.
- **Limites de l'Edge Runtime :** Timeouts potentiels lors de l'analyse de sites massifs.
- **Obsolescence des Schémas :** Évolution rapide des standards supportés par les LLMs (OpenAI, Google Gemini).

## 7. Signaux de Diagnostic
Pour identifier les anomalies de fonctionnement :
- **Requêtes SQL :** Monitoring de la table `machine_layer_scans` pour détecter les volumes d'erreurs ou les incohérences de statut.
- **Logs Edge :** Analyse des journaux d'exécution des fonctions Supabase pour isoler les erreurs de timeout ou de mémoire.
- **Debug Frontend :** Inspection des retours du hook `useMachineLayerStatus` dans la console développeur.
