### Signaux d'alerte globaux
- **SSRF double** : probe URL fournie par l'utilisateur sans denylist IP privée dans deux edges (register-api-key + patch-content).
- **api_key en clair** : aucun chiffrement au repos pour les secrets CMS malgré la mention dans le brief.
- **marked sans sanitizer** : sortie HTML poussée vers le CMS distant sans DOMPurify.
- **localStorage avec secrets** : bearerKey/appPassword persistés côté client.
- **cms_page_content en lecture publique totale** (`USING (true)`) — fuite de contenu cross-tenant.

## #187 · P1 · 2j — SSRF sur `site_url` dans cms-register-api-key

**Fichier** : `supabase/functions/cms-register-api-key/index.ts` L.327-355

La validation `new URL(userSiteUrl)` vérifie uniquement le format et le protocole (`http`/`https`). Aucun filtrage de l'hostname n'est appliqué avant `fetch(baseUrl + writeProbePath)`. Un utilisateur authentifié peut soumettre `site_url: "http://169.254.169.254/latest/meta-data/"` (cloud metadata AWS/GCP), `http://127.0.0.1:5432/` (PostgreSQL interne), ou toute IP RFC-1918 (`10.x`, `172.16.x`, `192.168.x`) et déclencher une requête sortante depuis l'edge Supabase avec sa clé API en header, transformant la fonction en proxy SSRF.

```
// L.327-337 — aucun contrôle d'hostname
const userSiteUrl = (body.site_url || `https://${site.domain}`).trim()
try {
  const u = new URL(userSiteUrl)
  if (u.protocol !== 'https:' && u.protocol !== 'http:') throw new Error('bad protocol')
} catch { ... }
baseUrl = remoteCfg!.resolveBaseUrl!(userSiteUrl)   // <-- feed directement vers fetch()
```

**Fix** : après `new URL(userSiteUrl)`, extraire `u.hostname`, rejeter si IP privée/loopback/link-local (`/^(127\.|10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.|169\.254\.|::1|localhost)/i.test(u.hostname)`) et vérifier que le hostname correspond au domaine du `tracked_site` enregistré (propriété prouvée).

## #188 · P1 · 1j — SSRF sur `target_url` dans cms-patch-content

**Fichier** : `supabase/functions/cms-patch-content/index.ts` L.1309-1322

L'enrichissement des injection points déclenche un `fetch(target_url)` où `target_url` est lu directement depuis le body de la requête, sans aucune validation d'hostname :

```typescript
const htmlResp = await fetch(target_url, {   // L.1315 — target_url du body utilisateur
  headers: { 'User-Agent': '...' },
  signal: AbortSignal.timeout(10000),
  redirect: 'follow',   // amplifie le risque (redirect ouvert vers IP interne)
})
```

L'option `redirect: 'follow'` aggrave le risque : un redirect HTTP 302 peut relayer vers une IP interne non atteignable directement par l'utilisateur. La fonction ne valide pas non plus que `target_url` correspond à un domaine appartenant à l'utilisateur (la vérification est faite sur `tracked_site_id` → `cms_connections` mais pas sur l'URL cible elle-même).

**Fix** : valider que `new URL(target_url).hostname` correspond au domaine du `tracked_site` lié à `tracked_site_id` ; ajouter la denylist IP privée identique à #187 ; remplacer `redirect: 'follow'` par `redirect: 'error'` ou valider le hostname post-résolution.

## #189 · P1 · 2j — api_key / oauth_tokens stockés en clair dans cms_connections

**Fichier** : `supabase/functions/cms-register-api-key/index.ts` (commentaire L.19), migrations `20260319*`

Le commentaire du code est explicite : `"api_key: stocké en clair (cohérent avec basic_auth_pass / oauth_*)"`. Les migrations confirment l'absence de tout `pgp_sym_encrypt`, `pgsodium.crypto_aead_det_encrypt` ou Supabase Vault. La protection existante (REVOKE SELECT depuis `authenticated`) est une défense PostgREST uniquement : un dump de la base, un accès service-role compromis, ou une fuite via une requête SQL mal construite dans une autre edge exposent l'intégralité des clés API de tous les clients en clair.

Colonnes concernées : `api_key`, `basic_auth_pass`, `oauth_access_token`, `oauth_refresh_token`.

**Fix** : utiliser `pgsodium` (disponible nativement sur Supabase) via `pgsodium.crypto_aead_det_encrypt(data, ad, key_id)` à l'écriture dans l'edge, et déchiffrer uniquement dans les edges service-role au moment du push ; ou migrer vers Supabase Vault (`vault.create_secret`/`vault.decrypted_secrets`). Créer une migration de rétro-chiffrement des lignes existantes.

## #190 · P1 · 1j — XSS stocké : sortie `marked` sans sanitisation dans dictadevi-actions

**Fichier** : `supabase/functions/dictadevi-actions/index.ts` L.56-60

```typescript
function convertMarkdownToHtml(md: string): string {
  marked.setOptions({ gfm: true, breaks: false })
  return (marked.parse(md, { async: false }) as string).trim()
  // ← aucun sanitizer appliqué
}
```

Le HTML produit est injecté directement dans les champs `content`/`body`/`excerpt` envoyés à l'API Dictadevi. Si le Markdown source contient des constructions forgées (`[XSS](javascript:alert(1))`, balises `<script>` intercalées dans du code-fence, etc.), le HTML généré est poussé tel quel dans le CMS. Dictadevi affiche ce contenu aux visiteurs → XSS stocké côté CMS client.

**Fix** : importer `linkedom` (disponible Deno) pour exposer un `DOMParser` côté serveur, puis passer la sortie de `marked` à `DOMPurify.sanitize(html, { ALLOWED_TAGS: [...], ALLOWED_ATTR: [...] })` avant toute assignation. Alternativement, utiliser l'option `marked-sanitizer-html` ou remplacer par `unified` + `rehype-sanitize`.

## #191 · P1 · 1j — cms_page_content en lecture publique totale (fuite cross-tenant)

**Fichier** : `supabase/migrations/20260423162641_*.sql` L.22-25

```sql
CREATE POLICY "Public read access to page content"
  ON public.cms_page_content FOR SELECT
  USING (true);   -- ← aucun filtre user_id / tenant
```

Toutes les lignes de `cms_page_content` sont lisibles sans authentification, depuis PostgREST ou l'API Supabase client. La table contient le contenu éditorial (H1, H2, FAQ, CTA, schema.org) de toutes les pages de tous les sites de tous les clients. Un concurrent ou un attaquant peut extraire l'intégralité du corpus de contenus Crawlers en une seule requête `GET /rest/v1/cms_page_content?select=*`. Le commentaire "Anyone can read (public marketing pages)" confond les pages marketing de Crawlers lui-même avec les pages des clients hébergées dans la table.

**Fix** : remplacer `USING (true)` par `USING (auth.uid() = user_id)` et ajouter une colonne `user_id` si absente ; si certaines lignes doivent rester publiques, ajouter une colonne `is_public boolean default false` et filtrer `USING (is_public = true OR auth.uid() = user_id)`.

## #192 · P2 · 1j — Absence de rate-limit sur cms-register-api-key

**Fichier** : `supabase/functions/cms-register-api-key/index.ts` (intégralité)

La fonction ne comporte aucun mécanisme de limitation de débit : pas de compteur par `user_id`, pas de délai minimum entre deux appels, pas de header `X-RateLimit-*`. Un utilisateur authentifié peut :
1. Boucler sur l'endpoint pour bruteforcer des formats de clés API.
2. Amplifier l'attaque SSRF (#187) en effectuant des milliers de probes vers des cibles internes.
3. Générer des coûts réseau significatifs (chaque appel déclenche deux `fetch()` vers le CMS cible).

**Fix** : insérer un enregistrement `cms_register_probes(user_id, last_probe_at)` et rejeter avec 429 si `now() - last_probe_at < 10s` ; ou utiliser Upstash Redis (déjà présent dans d'autres edges) avec un sliding-window de 10 appels / minute / user.

## #193 · P2 · 0.5j — Credentials CMS persistés en clair dans localStorage

**Fichier** : `src/components/Profile/SmartCmsConnectModal.tsx` L.113, L.175-197

```typescript
const credsStorageKey = siteDomain ? `cms-creds:${siteDomain}` : '';
// ...
localStorage.setItem(credsStorageKey, JSON.stringify(payload));
// payload contient : appPassword (WP App Password) et/ou bearerKey (dk_xxx)
```

Les secrets sont persistés en clair dans `localStorage` à la demande de l'utilisateur ("Se souvenir"). Tout script exécuté sur le domaine (XSS tiers, extension malveillante, injection pub) peut lire `localStorage` et exfiltrer les clés. Un WP App Password compromis donne un accès REST complet au site WordPress du client.

**Fix** : ne stocker qu'un booléen `{credsSaved: true}` dans localStorage pour pré-cocher la case ; les secrets ne doivent jamais quitter la session en cours. Si la persistance est un besoin UX fort, utiliser l'API `sessionStorage` (scope onglet uniquement) et documenter explicitement le risque à l'utilisateur avant activation.

## #194 · P2 · 1j — Plugin PHP généré avec `corrective_script` non échappé

**Fichier** : `src/components/Profile/WordPressConfigCard.tsx` L.64-68

```php
add_action('wp_footer', function() {
    $config = get_option('crawlers_config', array());
    if (!empty($config['corrective_script'])) {
        echo '<script>' . $config['corrective_script'] . '</script>' . "\n";
        // ← aucun wp_kses(), aucun esc_js(), injection directe
    }
});
```

Le plugin PHP proposé au téléchargement injecte `corrective_script` issu de l'API Crawlers directement dans le footer WP sans aucune validation côté WordPress. Si le compte Crawlers de l'utilisateur est compromis, ou si une edge Crawlers est compromise, un attaquant peut modifier `corrective_script` pour exécuter du JavaScript arbitraire sur le site du client (vol de session, redirection, skimming). La surface d'impact est maximale car le script est injecté sur toutes les pages.

**Fix** : remplacer `echo '<script>' . $config['corrective_script'] . '</script>'` par un mécanisme de nonce + CSP ou par l'injection d'un hash de script vérifié via signature HMAC côté Crawlers ; côté front, avertir explicitement l'utilisateur que le plugin accorde confiance totale à l'API Crawlers et afficher la clé utilisée pour vérifier l'intégrité.

## #195 · P3 · 1j — Prolifération de `any` dans cmsContentScanner.ts (7 occurrences)

**Fichier** : `supabase/functions/_shared/cmsContentScanner.ts` L.111, 133, 236, 254, 272, 327, 343

```typescript
const posts: any[] = Array.isArray(data) ? data : (data?.data?.posts || ...)
async function scanWordPress(conn: any, ...)   // L.236
async function scanShopify(conn: any, ...)     // L.272
async function scanDictadevi(conn: any, ...)   // L.327
```

Les signatures `any` désactivent toute vérification TypeScript sur les réponses CMS. Une réponse inattendue (pagination imbriquée, erreur JSON, changement d'API distante) peut provoquer un runtime panic (`Cannot read properties of undefined`) non capturé. Les fonctions de scan sont appelées depuis `universal-content-scanner` sans try/catch global sur chaque scanner individuel — une erreur non gérée fait échouer silencieusement l'inventaire complet.

**Fix** : déclarer des interfaces `WpPostRaw`, `ShopifyArticleRaw`, `DictadeviPostRaw` et utiliser `unknown` + type narrowing (`if (!Array.isArray(posts)) return`). Envelopper chaque appel `scanXxx()` dans son propre `try/catch` pour isoler les panics par plateforme.
