## #231 · P1 · 3j — DataForSEO + SerpAPI appelés directement hors gateway

**Localisation** : registry.ts L1365-1413 (`runDataForSEO`, `runSerpAPI`)

Les deux fonctions ouvrent un `fetch()` direct vers `api.dataforseo.com` et `serpapi.com` depuis la Edge Function, les credentials lus via `Deno.env.get()`. Aucun proxy interne, aucun circuit breaker, aucun plafond de coût quotidien côté infra. Une fuite de `DATAFORSEO_LOGIN` ou `SERPAPI_KEY` (log, trace, stack) permet une consommation non plafonnée sans alerte.

**Fix** : Router ces appels via une Edge Function gateway interne (`data-router`) dotée d'un AbortController et d'un compteur de dépense journalière (table `api_cost_ledger`). Migrer les credentials vers Supabase Vault, supprimer les env vars.

## #232 · P2 · 5j — Aucun cache SERP (serpapi_cache non implémenté)

**Localisation** : registry.ts L1441-1535 (handler `live_search`)

Le brief mentionne une table `serpapi_cache`. Elle n'existe ni dans le code ni dans les migrations accessibles. Chaque appel `live_search` — y compris des requêtes identiques sur la même journée — génère un appel API payant. Sur un quota Pro de 20/jour multiplié par N utilisateurs Pro, le coût marginal est non nul et croissant sans cache.

**Fix** : Créer `serpapi_cache (id, query_hash TEXT UNIQUE, source TEXT, results JSONB, created_at, expires_at)`. Avant tout appel, tenter `SELECT results WHERE query_hash = md5(query||mode||cc) AND expires_at > now()` ; stocker le résultat frais avec TTL 4h SERP / 24h Places.

## #233 · P1 · 2j — Race condition TOCTOU sur quota Pro live_search (20/jour)

**Localisation** : registry.ts L1301-1329 (`checkLiveSearchQuota`) + L1511-1522 (insert analytics)

Pour `agency_pro`, le quota est vérifié en lisant `analytics_events` (count), puis l'event est inséré après l'appel API. Les deux opérations ne sont pas atomiques. Deux requêtes parallèles sur la même session voient `used = 19 < 20`, passent toutes les deux, consomment deux crédits API et n'insèrent l'event qu'ensuite. Le dépassement est possible jusqu'à concurrence du nombre de connexions simultanées.

**Fix** : Créer une RPC Postgres `increment_and_check_quota(p_user_id uuid, p_event_type text, p_limit int, p_window_hours int) RETURNS bool` avec `INSERT ... ON CONFLICT DO UPDATE SET count = count + 1` ; retourner `count <= limit` de manière atomique.

## #234 · P2 · 2j — bumpSessionCounter non-atomique (Free / Premium)

**Localisation** : registry.ts L1351-1363 (`bumpSessionCounter`)

Le compteur session est mis à jour via un read-modify-write en mémoire TypeScript : on lit `context.live_search_count`, on incrémente, on réécrit le JSON. Deux appels simultanés (fenêtre parallèle de l'orchestrateur) lisent tous les deux `count = 1` et réécrivent `2`, absorbant un crédit au lieu de deux. Un utilisateur Free peut ainsi déclencher 6 recherches au lieu de 3 sous charge.

**Fix** : Remplacer par une RPC `increment_session_counter(p_session_id uuid, p_field text)` qui exécute `UPDATE copilot_sessions SET context = jsonb_set(context, '{live_search_count}', ((context->>'live_search_count')::int + 1)::text::jsonb) WHERE id = p_session_id`.

## #235 · P2 · 1j — market_diagnosis_page_targeting_v2 absent du registry comme skill autonome

**Localisation** : personas.ts (absence), registry.ts (intégration dans `market_diagnosis`)

Le brief liste `market_diagnosis_page_targeting_v2` comme skill distinct (matrice 8×8 séparée). En production il est embarqué dans `market_diagnosis` : même quota, même policy, aucune entrée dans `skillPolicies`, non invocable indépendamment. Cela interdit un appel ciblé de la matrice sur un site déjà diagnostiqué (sans relancer les 4 appels DataForSEO coûteux).

**Fix** : Extraire la logique `fetchPageSignals` + `decidePageVerdict` en SkillDefinition `market_diagnosis_page_targeting_v2` avec `required: ['tracked_site_id', 'keywords']` et quota propre (ex: 5/jour Pro, partagé avec `market_diagnosis`). Ajouter l'entrée `'auto'` dans les deux personas.

## #236 · P3 · 1j — Axe H hardcodé à 50, pondération nulle dans tous les profils métier

**Localisation** : registry.ts L1730 (`H = 50`) + L1622-1634 (`getBusinessModelWeights`)

Dans `decidePageVerdict`, `H = 50` est une constante. Tous les `getBusinessModelWeights` retournent `H: 0`. L'axe H est donc un dead letter documenté comme existant dans l'interface `AxisWeights` mais sans aucun effet sur le score. Il pollue la lisibilité de la matrice et trompe tout auditeur qui compte 8 axes réels.

**Fix** : Soit affecter H à un signal réel mesurable (ex: présence schema_org, canonical valide), soit supprimer l'axe H de `AxisWeights` et descendre à une matrice 7 axes honnête. Mettre à jour la description du skill en conséquence.

## #237 · P2 · 1j — Lookup tracked_site dans market_diagnosis via ilike('%domain%') ambigu

**Localisation** : registry.ts L1942-1946

```typescript
.eq('user_id', ctx.userId).ilike('domain', `%${domain}%`)
```

Un utilisateur possédant `foo.com` et `foobar.com` : une requête `market_diagnosis` sur `foo` peut retourner `foobar.com` en premier (ordre `created_at DESC`). Le workbench insert et le cooldown 90j s'appliquent alors au mauvais site. Le domaine est pourtant déjà normalisé ligne 1809-1810 — l'eq strict suffirait.

**Fix** : Remplacer l'ilike par `.eq('domain', domain)` strict. Si nécessaire, ajouter un fallback ilike uniquement lorsque la recherche exacte retourne null, avec un log warning.

## #238 · P2 · 3j — Numéro de téléphone stocké en clair dans sav_conversations

**Localisation** : registry.ts L2167 (`phone_callback: phone`)

Le numéro normalisé est inséré directement dans `sav_conversations.phone_callback` sans chiffrement. Seul le log analytics masque partiellement (`phone.slice(0,4) + '****' + phone.slice(-2)`). La table `sav_conversations` est accessible via service role par tout admin ; une extraction de la table (dump, requête BI) expose les numéros en clair, en violation du principe de minimisation RGPD.

**Fix** : Chiffrer `phone_callback` via `pgp_sym_encrypt(phone, current_setting('app.phone_key'))` à l'INSERT ; déchiffrer uniquement dans le dashboard admin via RPC dédiée. Après `phone_callback_expires_at`, écraser la colonne avec NULL via un cron job ou trigger.

## #239 · P1 · 2j — Race condition TOCTOU identique sur quota market_diagnosis

**Localisation** : registry.ts L1553-1578 (`checkMarketDiagnosisQuota`) + L2047-2060

Même vulnérabilité que #233 : `checkMarketDiagnosisQuota` lit le count `analytics_events` puis l'event est inséré post-exécution. Les 4 appels DataForSEO (L1828-1846) sont exécutés entre les deux, soit environ 2-4 secondes de fenêtre de race. Un utilisateur Free peut ainsi lancer 2 diagnostics (coût ~0.10$ chacun) au lieu d'1.

**Fix** : Même RPC atomique que #233 (`increment_and_check_quota`) avec `p_event_type = 'copilot_market_diagnosis'`. Bloquer avant les appels DFS, pas après.

## #241 · P2 · 3j — Pas de timeout sur les 4 appels DFS parallèles dans market_diagnosis

**Localisation** : registry.ts L1828-1846 (`Promise.all([dfsPost(...) × 4])`)

`dfsPost` ne dispose d'aucun `AbortController` ni timeout. Si DataForSEO répond en 30+ secondes (incident, throttling), l'Edge Function dépasse le timeout Supabase (60s par défaut) et est tuée. Le quota analytics_events n'est jamais incrémenté (bloqué après le Promise.all), mais les crédits DataForSEO sont consommés. En cas d'incident DFS récurrent, le quota apparent reste à 0 alors que les dépenses sont réelles.

**Fix** : Wrapper `dfsPost` avec `Promise.race([fetch(...), new Promise((_, r) => setTimeout(() => r(new Error('timeout')), 8000))])`. Ajouter un `Promise.race` global sur l'ensemble du handler à 25s. Loguer les timeouts dans `copilot_actions` avec `status = 'timeout'`.

## #242 · P3 · 2j — escalate_to_phone sans deduplication inter-sessions

**Localisation** : registry.ts L2152-2174 (création sav_conversation)

La deduplication n'existe qu'intra-session (via `sav_conversation_id` persisté dans `copilot_sessions.context`). Si l'utilisateur ouvre une nouvelle conversation, un nouveau `sav_conversations` est créé avec le même numéro. Il peut ainsi générer N tickets SAV actifs pour un seul blocage, saturant la file de rappel opérationnel.

**Fix** : Avant INSERT, vérifier `SELECT id FROM sav_conversations WHERE user_id = ctx.userId AND phone_callback = phone AND phone_callback_expires_at > now() LIMIT 1`. Si trouvé, mettre à jour et retourner l'existant plutôt que créer. Ajouter un index sur `(user_id, phone_callback_expires_at)`.

## #243 · P2 · 2j — Workbench insert dans market_diagnosis sans contrainte d'unicité (doublons)

**Localisation** : registry.ts L1997-2025 (`await ctx.service.from('architect_workbench').insert(rows)`)

Si `market_diagnosis` est invoqué deux fois sur le même domaine dans la journée (quota Pro = 10/jour), les mêmes URLs avec les mêmes verdicts sont insérées en double. Le cooldown 90j filtre les URLs déjà présentes par `created_at >= ninetyDays`, mais inclut les entrées fraîches du second appel — le doublon du premier appel est exclu uniquement si les deux appels sont espacés de moins d'une seconde.

**Fix** : Ajouter `ON CONFLICT (user_id, domain, target_url, source_function) DO UPDATE SET severity = EXCLUDED.severity, payload = EXCLUDED.payload, updated_at = now()` sur la table `architect_workbench` (nécessite une contrainte UNIQUE composite).

## #244 · P2 · 1j — Quota Pro lu via ctx.service sans filtre created_at borné côté client

**Localisation** : registry.ts L1314 (`since = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString()`)

Le calcul de la fenêtre glissante 24h est fait côté TypeScript, puis injecté comme paramètre `.gte('created_at', since)`. Si l'horloge du runtime Deno diverge de l'horloge Postgres (NTP drift), la fenêtre est décalée. Plus critique : `Date.now()` est évalué à chaque appel individuel ; sous forte charge, deux appels distants de quelques millisecondes peuvent avoir des fenêtres légèrement différentes.

**Fix** : Déplacer le calcul dans une RPC Postgres `check_quota_daily(p_user_id, p_event_type, p_limit)` qui utilise `now() - interval '24 hours'` côté serveur, garantissant une horloge unique et éliminant le drift.

## #245 · P3 · 1j — Personas : market_diagnosis_page_targeting_v2 absent des skillPolicies (forbidden silencieux)

**Localisation** : personas.ts L36-103 (FELIX_CONFIG, STRATEGIST_CONFIG)

Ni Félix ni le Stratège ne listent `market_diagnosis_page_targeting_v2` dans leurs `skillPolicies`. La `defaultSkillPolicy` étant `'forbidden'`, le skill serait refusé sans message explicite si le LLM tentait de l'appeler. Comme le skill n'existe pas non plus dans le registry (voir #235), l'erreur serait opaque : `skill non trouvé` plutôt que `forbidden`, court-circuitant le log de sécurité qui ne s'inscrit que pour les `rejected`.

**Fix** : Après création du skill autonome (#235), l'ajouter explicitement à `skillPolicies` des deux personas (`'auto'` pour Stratège, `'auto'` pour Félix). Documenter dans personas.ts que l'absence de clé = forbidden, afin que tout nouveau skill soit conscient de ce défaut fermé.
