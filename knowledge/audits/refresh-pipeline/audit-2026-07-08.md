## #200 · P1 · 1j — Coût LLM cumulé invisible : 5/6 skills hors aiGatewayFetch

**Constat.** `update-guidance` est le seul skill du pipeline qui passe par `aiGatewayFetch` (traçage `ai_gateway_usage`). Les autres skills LLM-intensifs contournent la gateway : `update-draft-consolidate` forward vers `editorial-pipeline-run` via un `fetch` direct sans propagation de traçage ; `update-claims-audit` appelle SerpAPI sans comptabilisation dans `ai_gateway_usage`. Le brief exige `SUM(cost_usd)` par skill — cette requête renverrait `NULL` pour 5/6 skills. La cible budgétaire du brief est structurellement non mesurable.

**Fix 1.** Envelopper l'appel `editorial-pipeline-run` dans `update-draft-consolidate` avec un wrapper qui logue tokens/coût dans `ai_gateway_usage` (skill = `draft`) avant de retourner la réponse.  
**Fix 2.** Ajouter dans `update-claims-audit` un appel `trackPaidApiCall('update-claims-audit', 'serpapi', 'search')` à chaque vérification, et consolider en fin de run un INSERT dans `ai_gateway_usage` avec le total de requêtes SerpAPI du cycle.

## #201 · P1 · 1j — Orchestration 100% séquentielle : claims + topic\_gaps + mentions parallélisables

**Constat.** Les 6 skills sont invoquées en chaîne stricte côté client (extract → claims → topic\_gaps → guidance → mentions → draft). Il n'existe aucun orchestrateur serveur. Or claims, topic\_gaps et mentions sont trois skills indépendantes qui ne nécessitent que l'artefact `extracted` — elles pourraient tourner en `Promise.all`. Sur un article de 2000 mots avec 3 concurrents et 8 claims, le temps total observé est ~45-60 s de latence réseau cumulée. `update-draft-consolidate` charge déjà les 4 stages en une seule requête SQL (`.in('stage', [...])`) mais c'est trop tard.

**Fix 1.** Créer un edge `update-orchestrate` qui reçoit `{ slug, competitor_urls, max_claims }` et exécute en parallèle `Promise.all([callClaims, callTopicGaps, callMentions])` après que `update-extract-content` a produit son artefact.  
**Fix 2.** En attendant l'orchestrateur, documenter l'ordre obligatoire dans le brief UI et exposer un bouton "Lancer l'analyse complète" qui fire les 3 fetches en parallèle depuis le frontend.

## #202 · P1 · 2j — Anti-hallucination gates absents : aucun blocage technique avant cms-patch-content

**Constat.** Ni `checkSemanticGate` ni `diagnose-hallucination` ne sont appelés dans la chaîne update. La directive `GUIDANCE_PROMPT` interdit d'inventer des faits mais c'est une contrainte prompt, pas un gate technique. `update-claims-audit` détecte les claims contredits (verdict = `contradicted`) et les inscrit dans l'artefact `claims`, mais rien ne bloque `update-draft-consolidate` ni `update-publish-draft` si des claims contredits existent. `hallucination_corrections` est alimentée par un flux séparé (`diagnose-hallucination`) non branché aux edges update-\*. L'objectif du brief (taux correction > 99 %) est inatteignable sans gate bloquant.

**Fix 1.** Dans `update-draft-consolidate`, vérifier avant l'appel `editorial-pipeline-run` que `claims.summary.contradicted === 0` ; sinon retourner HTTP 412 avec la liste des claims contredits à corriger manuellement.  
**Fix 2.** Dans `update-publish-draft`, appeler `diagnose-hallucination` sur le `content_html` du draft avant le patch CMS, et persister le résultat dans `hallucination_corrections` si des divergences sont détectées (seuil configurable via `update-config`).

## #203 · P2 · 0.5j — update-extract-content : monolithe standalone hors updatePipelineGuards

**Constat.** `update-extract-content` est le seul skill qui n'importe pas `updatePipelineGuards.ts`. Il réimplémente intégralement auth JWT + gating Premium+ inline (import direct `@supabase/supabase-js@2.45.0`, `PREMIUM_PLANS` redéfini localement avec 5 plans). Si `PREMIUM_PLANS` évolue dans le shared (ajout d'un plan `enterprise_plus` par exemple), la divergence sera silencieuse : le skill #1 de la chaîne rejettera les nouveaux plans alors que les skills #2-#6 les accepteront. Le fichier fait 220 lignes sans séparation concerns.

**Fix 1.** Remplacer les 60 premières lignes d'`update-extract-content` par `import { authAndGate } from '../_shared/updatePipelineGuards.ts'` et supprimer la redéfinition locale de `PREMIUM_PLANS`.  
**Fix 2.** Ajouter un test unitaire qui importe les deux définitions de `PREMIUM_PLANS` et vérifie leur égalité stricte pour détecter toute future dérive.

## #204 · P2 · 1j — update-market-trends : données de marché simulées en production (jitter aléatoire)

**Constat.** `fetchMonthlyMarketData()` retourne des valeurs hardcodées (`ChatGPT: 0.25`, `Gemini: 0.12`, etc.) auxquelles est appliqué un `jitter` pseudo-aléatoire `±0.03`. Un `TODO` commenté indique "In production, this would call external data APIs". Les artefacts `market_trends` persistés dans `update_artifacts` sont donc des fictions statistiques présentées comme des données réelles. Aucune mention de cette limitation dans la réponse API ni dans la doc.

**Fix 1.** Ajouter un champ `"data_source": "simulated"` dans le payload persisté et retourner un header `X-Data-Quality: simulated` tant que les APIs réelles (SimilarWeb, StatCounter) ne sont pas branchées.  
**Fix 2.** Bloquer l'affichage de ces métriques dans l'UI avec une bannière "Données indicatives – connexion API marché requise" jusqu'à intégration réelle.

## #205 · P2 · 0.5j — patch_effectiveness : GRANT admin-only, inaccessible au pipeline Premium+

**Constat.** La migration `20260404` crée deux policies sur `patch_effectiveness` : SELECT et INSERT réservées aux `has_role(auth.uid(), 'admin')`. Or `measure-patch-impact` utilise `getServiceClient()` (service role) qui bypass RLS — cela fonctionne pour l'admin interne. Mais si le pipeline refresh tente de logguer l'efficacité d'un patch CMS effectué par un user Premium+ (via `update-publish-draft`), l'INSERT échouera en RLS. La table n'a pas de colonne `user_id` ce qui empêche une policy user-scoped directe.

**Fix 1.** Ajouter une colonne `user_id UUID` nullable à `patch_effectiveness` + une policy `INSERT FOR authenticated WITH CHECK (user_id = auth.uid() OR public.has_role(auth.uid(), 'admin'))`.  
**Fix 2.** Faire écrire `update-publish-draft` dans `patch_effectiveness` via `getServiceClient()` en passant `user_id` explicitement, en attendant la migration de la policy.

## #206 · P2 · 0.5j — content\_monitor\_log : politique INSERT absente pour authenticated

**Constat.** La migration `20260330` crée une seule policy sur `content_monitor_log` : `SELECT` pour `authenticated WHERE user_id = auth.uid()`. Aucune policy INSERT, UPDATE ou DELETE. Toute edge qui tente d'écrire un log de monitoring sous JWT user (ex. `cms-patch-content` loggant le résultat du patch) reçoit un RLS 403 silencieux. L'autopilot-engine écrit via `getServiceClient()` (service role), ce qui masque le problème.

**Fix 1.** Ajouter `CREATE POLICY "Users insert own monitor logs" ON public.content_monitor_log FOR INSERT TO authenticated WITH CHECK (user_id = auth.uid())`.  
**Fix 2.** Auditer `postExecute.ts` pour confirmer que tous les INSERTs dans `content_monitor_log` passent par service role et documenter ce choix explicitement dans la migration.

## #207 · P2 · 1j — cms-patch-content : accumulation de timeouts CMS, pas de garde globale 60s

**Constat.** `cms-patch-content` enchaîne pour chaque plateforme : GET du post (15s), PATCH (30s), vérification post-patch (15s), soit jusqu'à 60s par cycle — exactement la limite de timeout standard Supabase Edge Functions. Pour Webflow, la séquence GET collections (15s) + GET item (10s) + PATCH (30s) dépasse déjà 55s avant vérification. Aucun `AbortController` global sur l'edge. Si une plateforme CMS est lente, l'edge sera tué sans retour au client, le patch CMS sera partiellement appliqué et non loggué.

**Fix 1.** Ajouter en tête de `Deno.serve` un `AbortController` global avec `setTimeout(() => controller.abort(), 55_000)` et retourner HTTP 504 structuré si déclenché, permettant au client de relancer.  
**Fix 2.** Persister l'état du patch dans `content_monitor_log` avant chaque appel CMS (status='in_progress') pour permettre une reprise idempotente si l'edge est tué.

## #208 · P3 · 0.5j — update-guidance : modèle LLM inexistant (google/gemini-3-flash-preview)

**Constat.** `update-guidance` appelle `aiGatewayFetch` avec `model: 'google/gemini-3-flash-preview'`. Ce slug de modèle n'existe pas dans OpenRouter ni dans la Lovable AI Gateway (le modèle correct est `google/gemini-flash-1.5` ou `google/gemini-2.0-flash-preview:thinking`). En cas d'erreur 404/422 du gateway, le catch attrape l'exception et retourne HTTP 502 `llm_failed` — le skill guidance échoue silencieusement, bloquant l'ensemble de la chaîne sans artefact `guidance`.

**Fix 1.** Corriger le slug en `google/gemini-flash-1.5` (stable) et ajouter un fallback sur `openai/gpt-4o-mini` si le premier modèle retourne 4xx.  
**Fix 2.** Ajouter un test d'intégration qui valide la disponibilité du modèle au démarrage de l'edge (health-check via `aiGatewayFetch` avec `max_tokens: 1`).

## #209 · P3 · 0.5j — update\_artifacts : expires\_at non enforced, aucune purge automatique

**Constat.** La migration `20260502` crée un champ `expires_at TIMESTAMPTZ DEFAULT (now() + interval '30 days')` et un index `update_artifacts_expires_idx`. Aucune migration ne crée de `pg_cron` ni de trigger de suppression automatique. Après 30 jours, les artefacts expirent conceptuellement mais restent en base. Sur une base avec 500 users Premium+ chacun traitant 10 URLs/mois, cela représente ~30 000 rows/mois non purgées, avec payloads JSONB volumineux (HTML extrait, claims, draft complet).

**Fix 1.** Créer une migration ajoutant `SELECT cron.schedule('purge-expired-update-artifacts', '0 3 * * 0', $$ DELETE FROM public.update_artifacts WHERE expires_at < now() $$)`.  
**Fix 2.** Ajouter à `update-deferred-tasks` une tâche de type `purge_expired_artifacts` qui exécute la même DELETE via service role pour les environnements sans pg_cron.
