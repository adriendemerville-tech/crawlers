**Périmètre :** edges `gmb-*`, `gbp-auth` + tables `gmb_locations`, `gmb_performance`, `gmb_posts`, `gmb_reviews`, `gmb_tracked_keywords`, `gmb_power_snapshots`, `gmb_local_competitors`  
**Findings :** #271 – #285  
**Criticité :** P1 = bloquant production · P2 = correctif sprint · P3 = dette technique

## #271 · P1 · 3j — RLS sans dimension tenant : isolation client-à-client absente en mode Pro Agency

Toutes les tables GMB isolent les lignes par `user_id = auth.uid()`. En mode Pro Agency, l'agence est un seul compte (`user_id`) gérant N fiches clientes : soit elle stocke toutes les `gmb_locations` sous son `user_id` (pas d'isolation entre clients A et B), soit chaque client possède son propre `user_id` (impossibilité de vue agrégée agence). Aucune colonne `agency_id` / `client_id` / `tenant_id` n'existe dans le schéma. Le risque de cross-tenant read entre clients d'une même agence est structurel.

**Fix :** Ajouter une colonne `agency_user_id UUID REFERENCES auth.users(id)` sur `gmb_locations` (nullable pour les comptes solo) ; refondre les policies en `USING (user_id = auth.uid() OR agency_user_id = auth.uid())` et créer un index composite `(agency_user_id, id)` pour la vue agrégée.

## #272 · P1 · 1j — `founderGmb.ts` : query `gmb_locations` cross-tenant sans filtre `user_id`

Dans `detectGoogleMyBusiness()` (`_shared/founderGmb.ts` l. 141), la recherche en base est `sb.from('gmb_locations').select(...).or('website.ilike.%domain%').limit(1)` via `service_role` sans clause `.eq('user_id', ...)`. N'importe quelle fiche d'un autre tenant qui partage un sous-domaine ou un domaine parent peut matcher : fuite de `location_name`, `address`, `category` vers un audit appartenant à un tiers.

**Fix :** Passer `user_id` en paramètre de `detectGoogleMyBusiness()` et ajouter `.eq('user_id', userId)` avant le `.or()` ; si le contexte est cross-user (audit externe), restreindre la recherche à la table publique DataForSEO uniquement.

## #273 · P2 · 1j — `gmb_power_snapshots` : policy INSERT mal étiquetée `TO authenticated` au lieu de `service_role`

La policy `"Service can insert gmb power snapshots"` est déclarée `TO authenticated` alors que l'edge `gmb-optimization` utilise `getServiceClient()` (service_role) qui bypasse RLS. Conséquence : (1) la policy ne s'applique jamais — service_role ignore toutes les policies ; (2) un client `authenticated` peut injecter un snapshot arbitraire pour n'importe quel `tracked_site_id` tant que `user_id = auth.uid()` passe le `WITH CHECK`. Le nom "Service can insert" crée une fausse confiance dans les revues de sécurité.

**Fix :** Changer le rôle de la policy en `TO service_role` et ajouter une policy `INSERT TO authenticated WITH CHECK (user_id = auth.uid() AND tracked_site_id IN (SELECT id FROM tracked_sites WHERE user_id = auth.uid()))` si les clients doivent écrire directement.

## #274 · P2 · 0.5j — `gmb_local_competitors` : pas de policy INSERT pour `authenticated`, asymétrie RLS silencieuse

La table `gmb_local_competitors` expose une policy SELECT pour `authenticated` et une policy ALL pour `service_role`. L'edge `gmb-local-competitors` contourne via `getServiceClient()` — les écrits bypasse le RLS sans log. Si un jour un appel direct depuis le frontend (SDK) tente un INSERT, il sera refusé sans message explicite. L'asymétrie n'est pas documentée et peut masquer une régression lors d'une refacto vers un client Supabase authentifié.

**Fix :** Ajouter une policy `INSERT TO authenticated WITH CHECK (user_id = auth.uid() AND gmb_location_id IN (SELECT id FROM gmb_locations WHERE user_id = auth.uid()))` ; documenter explicitement que l'edge utilise service_role par design.

## #275 · P2 · 1j — Refresh token GBP partagé avec les credentials GSC : couplage fort sans alerting

`gbp-auth` et `gmb-actions` utilisent `GOOGLE_GSC_CLIENT_ID` + `GOOGLE_GSC_CLIENT_SECRET` (variables nommées GSC) pour rafraîchir les access tokens GBP. Google peut révoquer le refresh_token GBP lors d'une rotation de secret GSC ou d'un changement de scope OAuth. En cas d'invalidation, les connexions GBP tombent silencieusement (`console.warn` uniquement) sans alerting ni re-auth guidée. Un client Pro Agency avec 50 fiches perd sa synchronisation sans notification.

**Fix :** Dédier des variables `GOOGLE_GBP_CLIENT_ID` / `GOOGLE_GBP_CLIENT_SECRET` distinctes ; ajouter un champ `last_refresh_error` sur `google_connections` et déclencher un email/webhook si le refresh échoue deux fois consécutives.

## #276 · P2 · 1j — `gbpFetch()` sans retry ni lecture `Retry-After` : quota 429 non géré

La fonction `gbpFetch()` (gmb-actions l. 199–211) lève immédiatement une exception sur tout status HTTP non-2xx, y compris 429 `RESOURCE_EXHAUSTED`. Google Business Profile API impose des quotas journaliers stricts (ex. 1 000 req/jour pour `getDailyMetricsTimeSeries`). En Pro Agency avec N locations, le quota peut être épuisé en quelques syncs. Aucun backoff exponentiel, aucune lecture du header `Retry-After`, aucune mise en queue.

**Fix :** Wrapper `gbpFetch()` avec retry conditionnel sur 429 (max 3 tentatives, délai = `Retry-After` ou `2^n` secondes) ; logger les 429 dans `analytics_events` avec `event_type: 'api:gbp:quota_exceeded'` pour monitoring.

## #277 · P2 · 0.5j — `gmb-places-autocomplete` : aucune authentification, exposition de coût Places API

L'edge `gmb-places-autocomplete` n'effectue aucune validation JWT (`handleRequest` sans `getAuthenticatedUser`). Chaque requête génère 1 appel Autocomplete (~$2.83/1 000) + jusqu'à 5 appels Place Details (~$17/1 000 en tarif 2026). Un acteur malveillant peut scraper massivement sans friction, consommant le budget Places API sans aucun rate-limit par utilisateur.

**Fix :** Ajouter `getAuthenticatedUser(req)` en tête de handler ; implémenter un rate-limit par `user_id` en base (`analytics_events` count sur fenêtre 1h) ou via un middleware Supabase Edge.

## #278 · P3 · 1j — `gmb_tracked_keywords` : absence de FK vers `gmb_locations`, risque d'orphelins

La table `gmb_tracked_keywords` référence `tracked_sites(id)` mais n'a aucune FK vers `gmb_locations`. Un keyword peut être associé à un site sans fiche GMB correspondante, ou la fiche peut être supprimée (cascade sur `tracked_sites`) en laissant des keywords orphelins avec des `current_position` / `previous_position` sans sens. En Pro Agency avec rotation de fiches clients, la dette de données est cumulable.

**Fix :** Ajouter `gmb_location_id UUID REFERENCES gmb_locations(id) ON DELETE SET NULL` (nullable pour les mots-clés génériques) ; créer un job de nettoyage hebdomadaire pour supprimer les keywords dont `tracked_site_id IS NULL AND gmb_location_id IS NULL`.

## #279 · P2 · 2j — `gbp-auth` action=login : `user_id` accepté du body sans validation JWT

Dans `gbp-auth` (l. 43–47), la branche `action === 'login'` assigne directement `user_id = body_user_id` sans valider le token Authorization. N'importe quel appelant non authentifié peut initier un flow OAuth GBP pour un `user_id` arbitraire — suffisant pour déclencher un `gsc-auth` avec `modules: ['gsc', 'gmb']` au nom d'un utilisateur tiers, potentiellement en remplacement de ses tokens.

**Fix :** Pour `action=login`, exiger soit un JWT valide (`getUserClient(authHeader).auth.getUser()`), soit un `state` signé HMAC côté frontend ; rejeter toute requête `login` sans Authorization valide.

---

## #280 · P3 · 1j — `console.log()` en production avec données métier sensibles

Les edges `gbp-auth`, `gmb-actions` et `gmb-local-competitors` émettent des `console.log` contenant `accountId`, `locationId`, `access_token` (implicite via refresh confirmé), search queries utilisateurs, et comptage de locations. En production Supabase, ces logs apparaissent dans le dashboard et peuvent être exportés dans des outils APM tiers ou des exports de conformité RGPD.

**Fix :** Remplacer les `console.log` contenant des identifiants ou données client par des logs structurés avec niveau `debug` désactivé en production ; ne jamais logger `access_token`, `refresh_token`, ni `accountId` complet — utiliser des extraits de 4 derniers caractères.

## #281 · P3 · 1j — `gbp-auth/status` : pas de cache, N+1 API Google à chaque appel

L'action `status` exécute systématiquement : 1 requête SQL `google_connections`, 1 refresh token si expiré, puis conditionnellement 2 appels API Google (accounts + locations). En mode Pro Agency avec un dashboard qui poll toutes les 30s, et N clients à l'écran, le nombre d'appels GBP API peut saturer le quota journalier. Aucun TTL en base, aucune couche de cache Redis/Upstash.

**Fix :** Ajouter une colonne `locations_discovered_at TIMESTAMPTZ` sur `google_connections` et ne relancer la discovery que si `now() - locations_discovered_at > interval '1 hour'` ; pour le refresh token, utiliser un seuil de 5 min avant expiration plutôt qu'à l'expiration exacte.

## #282 · P2 · 1j — Agrégation coûts DataForSEO sans circuit-breaker ni budget cap

Dans `gmb-local-competitors`, `trackPaidApiCall()` est appelé APRÈS la requête DataForSEO (l. 72). Il n'y a aucune vérification de quota ou budget avant l'appel. En Pro Agency, un agent déclenchant un scan simultané pour 50 fiches multiplie les appels sans frein. SerpAPI est utilisé en fallback sans tracking de coût (`trackPaidApiCall` absent sur la branche SerpAPI). Le coût réel est sous-estimé dans les rapports.

**Fix :** Déplacer `trackPaidApiCall` avant l'appel réseau (intent log) ; ajouter un check de budget mensuel via une fonction RPC `check_api_budget('gmb-local-competitors')` avant chaque scan ; ajouter `trackPaidApiCall` sur la branche SerpAPI.

## #283 · P3 · 2j — `any` TypeScript massif dans `gmb-actions` : 15+ occurrences, sûreté de type inexistante

`gmb-actions` contient plus de 15 annotations `any` explicites : `sb: any`, `conn: any`, `result: any`, `allLocations: any[]`, `formatHours(hours: any): Record<string, any>`, etc. Les réponses des APIs Google Business Profile ne sont pas typées — un changement de format API (ex. renommage `phoneNumbers` → `phones`) passe silencieusement en production et produit des `undefined` stockés en base sans erreur de compilation.

**Fix :** Créer un fichier `types/gbp-api.ts` avec les interfaces GBP (Account, Location, TimeSeries, Review) ; remplacer les `any` par des types stricts ou `unknown` avec guards ; activer `"strict": true` dans le `tsconfig` des fonctions Edge.

---

## #284 · P3 · 0.5j — `list-locations` : boucle `do/while` sans limite d'itérations, risque timeout/OOM

L'action `list-locations` dans `gmb-actions` (l. 480–500) pagine les locations GBP via une boucle `do { ... } while (nextPageToken)` sans compteur de guard ni limite maximale. Un compte agence avec plusieurs centaines de fiches (légal sur GBP) peut dépasser les 50 secondes de timeout Edge Function Supabase ou saturer la mémoire avec `allLocations: any[]` non bornée.

**Fix :** Ajouter `let pageCount = 0; const MAX_PAGES = 20;` avec `if (++pageCount > MAX_PAGES) break;` ; paginer côté client si nécessaire et retourner un `has_more: true` avec le dernier `pageToken` pour une pagination lazy.

---

## #285 · P3 · 0.5j — GRANT explicites absents sur `gmb_reviews`, `gmb_posts`, `gmb_performance`

Les migrations des tables `gmb_reviews`, `gmb_posts`, `gmb_performance` activent le RLS et créent des policies `TO authenticated` mais n'émettent aucun `GRANT SELECT/INSERT/UPDATE/DELETE ON TABLE public.gmb_* TO authenticated`. La sécurité repose sur le défaut Supabase (propriétaire = `postgres`, grants hérités). Lors d'une migration de schéma avec `ALTER TABLE ... OWNER TO` ou d'une réplication cross-environment, les grants implicites peuvent être perdus, rendant les tables illisibles côté client sans message d'erreur RLS intelligible.

**Fix :** Ajouter en fin de chaque migration GMB : `GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE public.gmb_reviews TO authenticated; GRANT ALL ON TABLE public.gmb_reviews TO service_role;` (idem pour `gmb_posts`, `gmb_performance`) ; valider avec `\z gmb_*` en post-migration CI.
