# Audit Brief - Pipeline Ingestion Analytics Google - 2026-07-08

## 1. Contexte et Objectifs
Cet audit porte sur la robustesse et l'efficience du pipeline d'ingestion des données Google (Search Console et Analytics 4). L'objectif est de valider la transition vers l'authentification unifiée et d'évaluer la scalabilité des processus de synchronisation face aux limites de quotas Google.

## 2. Périmètre Technique

### Edges (Crawlers & Connectors)
- gsc-auth: Gestion de l'authentification OAuth unifiée (mem://auth/google-oauth-unified-fr).
- gsc-sync-*: Synchronisation des données GSC (fetch-gsc-daily).
- ga4-sync-*: Récupération des métriques GA4 (fetch-ga4-data).
- gsc-bigquery-*: Interface avec BigQuery pour l'historisation longue (gsc-bigquery-query).
- google-ads-*: Connecteur en fin de vie (déprécié), maintenance minimale uniquement.

### Schéma de Données (Tables)
- Analytics: ga4_daily_metrics, ga4_top_pages, ga4_behavioral_metrics, ga4_traffic_sources_cache, ga4_history_log, ga4_page_groups.
- Search Console: gsc_daily_positions, gsc_history_log, gsc_bigquery_cache, gsc_bigquery_config.
- Système: google_connections, google_ads_history_log.

## 3. Architecture des Flux
L'architecture repose sur les standards d'intégration v5 (mem://tech/architecture/data-integration-v5-fr). 
1. Authentification centralisée via gsc-auth.
2. Déclenchement périodique des sync-edges pour l'ingestion quotidienne.
3. Mise en cache intermédiaire dans les tables locales avant historisation géolocalisée (mem://tech/analytics/geo-historization-fr).
4. Export vers BigQuery pour le stockage à long terme et les requêtes complexes.

## 4. Qualité, Performance et Quotas
La gestion des quotas Google API est critique. Les limites par projet et par utilisateur doivent être monitorées strictement pour éviter les interruptions de service.
Actuellement, les coûts BigQuery ne sont pas tracés dans la table ai_gateway_usage. Il est fortement recommandé d'implémenter une instrumentation spécifique pour imputer ces coûts par utilisateur ou par projet, alignant ainsi le monitoring BigQuery sur le reste de l'infrastructure IA.
Respect des standards d'intégration: mem://auth/google-integration-standards-fr.

## 5. Benchmark et Positionnement
Le pipeline actuel se compare aux solutions leaders du marché:
- SEranking: Performance équivalente sur la rapidité de synchronisation, mais SEranking propose une visualisation plus riche des clusters de mots-clés.
- Semrush Position Tracking: Avantage sur la granularité géographique; notre pipeline doit finaliser l'implémentation de geo-historization-fr pour combler l'écart.
- TODO Utilisateur: Intégration de la détection d'anomalies sur les chutes brutales de trafic.

## 6. Risques et Points de Vigilance
- Expiration des Refresh Tokens: Risque de déconnexion massive si le mécanisme de rotation de gsc-auth échoue.
- Changements d'API Google: GA4 et GSC API évoluent fréquemment, nécessitant une veille active sur les schémas de réponse.
- Coûts BigQuery: Croissance non linéaire des coûts sans politique de rétention ou d'agrégation agressive.

## 7. Outils de Diagnostic et Monitoring

### Requêtes SQL de Contrôle
Surveillance du taux de rotation des tokens et état des connexions:
```sql
SELECT 
  id, 
  user_id, 
  created_at, 
  updated_at, 
  (updated_at - created_at) as token_lifetime
FROM google_connections 
ORDER BY updated_at DESC;
```

### Analyse des Logs
Les logs des edges (ga4-sync, gsc-sync) doivent être filtrés pour identifier les erreurs 403 (Quota Exceeded) et 401 (Unauthorized):
- Monitoring via dashboard Edge Functions.
- Alerte si le taux d'échec dépasse 5% sur une fenêtre de 15 minutes.
