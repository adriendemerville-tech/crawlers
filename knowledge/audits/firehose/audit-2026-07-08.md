La chaîne de traitement expose deux risques majeurs : (1) le `tap_token_encrypted` est stocké **en clair** dans PostgreSQL malgré son nom ; (2) `firehose_events` croît sans borne, sans contrainte de déduplication ni politique de rétention. Les risques de saturation DB lors d'un pic SSE sont réels et non mitigés.

## #246 · P0 · 1j — tap_token_encrypted stocké en clair

**Localisation:** `supabase/functions/firehose-actions/index.ts` — fonctions `listTaps()` et `createTap()`  
**Constat:** La colonne `tap_token_encrypted` porte un nom trompeur : aucun chiffrement n'est appliqué avant persistance. Le token brut de l'API Firehose est inséré directement (`tap_token_encrypted: tap.token` / `tap_token_encrypted: token`). Un dump DB ou un accès service_role compromis expose tous les tokens en clair. La migration `20260525103510` révoque bien le SELECT côté `authenticated`, mais ne protège pas contre une exfiltration interne.

**Fix:**  
1. Chiffrer avec `pgcrypto.pgp_sym_encrypt(token, app.settings.token_encryption_key)` avant INSERT, déchiffrer à la lecture dans `getTapToken()`.  
2. Ajouter un secret `TOKEN_ENCRYPTION_KEY` en variable d'environnement Supabase et documenter la rotation.

## #247 · P1 · 2j — Absence de déduplication des événements (kafka_offset)

**Localisation:** `pollStream()`, ligne `sb.from('firehose_events').insert(inserts)`  
**Constat:** L'appel `poll_stream` peut être déclenché deux fois avec le même paramètre `since` (retry réseau, double cron, race condition frontend). Aucune contrainte `UNIQUE` sur `(tap_id, kafka_offset)` ni clause `ON CONFLICT DO NOTHING` dans l'INSERT. Des rafales SSE peuvent injecter des doublons indétectables qui faussent les dashboards et gonflent le stockage.

**Fix:**  
1. Ajouter en migration : `ALTER TABLE firehose_events ADD CONSTRAINT uq_firehose_tap_kafka UNIQUE (tap_id, kafka_offset);`  
2. Remplacer `.insert(inserts)` par `.upsert(inserts, { onConflict: 'tap_id,kafka_offset', ignoreDuplicates: true })`.

## #248 · P1 · 1j — Absence de politique de rétention (TTL) sur firehose_events

**Localisation:** Schéma `firehose_events`, migrations, aucun job pg_cron dédié  
**Constat:** Aucune colonne `expires_at`, aucune fonction de purge, aucun job pg_cron ciblant `firehose_events`. Avec un flux SSE actif (plusieurs taps, règles larges), la table peut accumuler des millions de lignes/semaine. Le reste du projet dispose de patterns de purge (copilot-purge-actions, TTL audit_cache) mais ils n'ont pas été appliqués à cette table.

**Fix:**  
1. Ajouter `expires_at timestamptz NOT NULL DEFAULT now() + interval '30 days'` à `firehose_events` et un index `CREATE INDEX ON firehose_events(expires_at)`.  
2. Créer un job pg_cron : `SELECT cron.schedule('purge-firehose-events', '0 3 * * *', $$DELETE FROM firehose_events WHERE expires_at < now()$$);`

## #249 · P1 · 0.5j — INSERT policy absente sur firehose_events (seul service_role écrit)

**Localisation:** Migration `20260323120753`, table `firehose_events`  
**Constat:** RLS est activé sur `firehose_events` avec des policies SELECT uniquement. Il n'existe pas de policy INSERT/UPDATE/DELETE pour `authenticated`. Les écritures passent exclusivement par `getServiceClient()` (service_role), ce qui est correct. Cependant, l'absence de policy INSERT explicite `WITH CHECK (false)` pour `authenticated` laisse une surface d'attaque : si un appel direct Supabase JS utilise accidentellement le client user au lieu du service client, l'INSERT est refusé par RLS mais le message d'erreur n'est pas catégoriquement documenté. Plus critique : `firehose_taps` accepte INSERT `authenticated` alors que la création devrait être réservée à la Edge Function.

**Fix:**  
1. Ajouter `CREATE POLICY "Block direct insert firehose_events" ON firehose_events FOR INSERT TO authenticated WITH CHECK (false);` pour documenter et verrouiller explicitement.  
2. Supprimer la policy `"Users can insert own firehose taps"` sur `firehose_taps` — la création de tap doit passer uniquement par la Edge Function via service_role.

## #250 · P2 · 1j — N+1 INSERTs dans listRules() (boucle synchrone)

**Localisation:** `listRules()`, boucle `for (const rule of rules) { await sb.from('firehose_rules').insert(...) }`  
**Constat:** La synchronisation des règles exécute un INSERT individuel par règle dans une boucle `await` séquentielle. Pour un tap avec 50 règles, cela génère 50 roundtrips DB séquentiels, bloquant l'isolate Deno pendant toute la durée et consommant des connexions Supabase pooler inutilement. Même pattern observé dans `createRule()` pour le recalcul de `rules_count` (2 requêtes imbriquées).

**Fix:**  
1. Extraire les nouvelles règles en amont, puis faire un seul `sb.from('firehose_rules').upsert(newRules, { onConflict: 'rule_id,tap_id', ignoreDuplicates: true })`.  
2. Remplacer le recalcul inline de `rules_count` par `SELECT count(*) FROM firehose_rules WHERE tap_id = $1` dans une RPC dédiée ou via un trigger `AFTER INSERT/DELETE`.

## #251 · P2 · 1j — Aucun rate-limit sur l'action poll_stream

**Localisation:** Router `Deno.serve`, case `'poll_stream'`, aucun middleware de throttling  
**Constat:** Un utilisateur `agency_pro` authentifié peut appeler `poll_stream` en boucle sans limitation. Chaque appel déclenche une connexion SSE vers `api.firehose.com` (30s timeout), une lecture complète du buffer texte, et un batch INSERT. À 1 req/s, cela représente ~86 400 appels/jour, saturant le pool de connexions Supabase et l'IOPS disque. Aucun header `Retry-After`, aucune table de rate-limit, aucun token bucket.

**Fix:**  
1. Ajouter une vérification `last_polled_at` sur `firehose_taps` : rejeter avec 429 si `now() - last_polled_at < interval '10 seconds'`.  
2. Mettre à jour `last_used_at` (déjà présent) comme timestamp de dernière poll et vérifier côté Edge Function avant d'appeler l'API upstream.

## #252 · P2 · 0.5j — diff_chunks JSONB non borné dans firehose_events

**Localisation:** `pollStream()`, ligne `diff_chunks: doc.diff?.chunks`  
**Constat:** Le champ `diff_chunks` est inséré sans troncature ni validation de taille. Un document avec un diff massif (page entière modifiée) peut injecter plusieurs Mo de JSONB dans une seule ligne. Sur un flux actif de 100 événements/poll, cela représente potentiellement des centaines de Mo par exécution. Le champ `markdown_excerpt` est lui borné à 2 000 caractères (`doc.markdown?.slice(0, 2000)`), mais `diff_chunks` ne bénéficie pas du même traitement.

**Fix:**  
1. Tronquer avant insertion : `diff_chunks: limitJsonbSize(doc.diff?.chunks, 50_000)` (50 ko max, helper à créer).  
2. Ajouter une contrainte de vérification SQL : `ALTER TABLE firehose_events ADD CONSTRAINT chk_diff_chunks_size CHECK (octet_length(diff_chunks::text) < 102400);`

## #253 · P2 · 0.5j — Index composite (tap_id, matched_at DESC) absent

**Localisation:** Migration `20260323120753`, indexes `firehose_events`  
**Constat:** Les requêtes dashboard filtrent systématiquement par `tap_id` puis trient par `matched_at DESC`. Les index existants sont séparés (`idx_firehose_events_tap_id` et `idx_firehose_events_matched_at`), forçant PostgreSQL à utiliser un index scan sur `tap_id` puis un sort explicite en mémoire. Sur une table à plusieurs millions de lignes, cela génère des seq scans coûteux. Un index composite couvrant les deux colonnes élimine le sort.

**Fix:**  
1. Ajouter en migration : `CREATE INDEX idx_firehose_events_tap_matched ON firehose_events(tap_id, matched_at DESC);`  
2. Supprimer `idx_firehose_events_matched_at` si les queries ne trient jamais sans filtre `tap_id` (analyser avec `EXPLAIN ANALYZE` avant suppression).

## #254 · P3 · 1j — Cast `any` systématique sans validation de payload upstream

**Localisation:** `pollStream()`, `listTaps()`, `createTap()`, `createRule()` — `(data as any)?.data`, `any[]`  
**Constat:** L'intégralité des réponses de l'API Firehose est consommée via des casts `as any` sans schéma de validation (pas de Zod, pas d'interface TypeScript stricte). Si l'API upstream change son contrat (champ renommé, type modifié), les erreurs sont silencieuses et propagées en DB sous forme de `null` ou de données malformées. La fonction `pollStream` construit des objets `events` à partir de champs non validés, dont `kafka_offset: parseInt(eventId.split('-')[1])` qui retourne `NaN` si le format change.

**Fix:**  
1. Définir des interfaces TypeScript strictes pour `TapResponse`, `RuleResponse`, `StreamEvent` et les appliquer après parsing.  
2. Valider `kafka_offset` : `const kafkaOffset = eventId ? (parseInt(eventId.split('-')[1]) || null) : null` — déjà partiellement fait mais NaN non géré explicitement.

## #255 · P3 · 0.5j — console.error expose des messages d'erreur DB bruts

**Localisation:** `createTap()`, `pollStream()`, router catch-all — `console.error('[firehose] Events insert error:', insertErr.message)`, `console.error('[firehose-actions] Error:', msg)`  
**Constat:** Les messages d'erreur PostgreSQL bruts (contraintes, noms de colonnes, détails de schéma) sont loggés via `console.error` dans les isolates Deno. Sur Supabase, ces logs sont accessibles via le dashboard Edge Function Logs. En cas d'accès non autorisé au dashboard (compte compromis), les logs révèlent la structure interne du schéma. De plus, dans le catch-all, `msg` est renvoyé tel quel dans la réponse HTTP 500 (`return err(msg, 500)`), exposant potentiellement des détails internes au client.

**Fix:**  
1. Dans le catch-all : remplacer `return err(msg, 500)` par `return err('Internal server error', 500)` et logger `msg` uniquement côté serveur.  
2. Introduire un wrapper `safeLog(level, context, error)` qui tronque les messages à 200 caractères et masque les patterns de schéma SQL (`relation "`, `column "`, `constraint "`).
