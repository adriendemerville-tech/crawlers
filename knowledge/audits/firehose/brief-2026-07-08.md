# Audit Brief: Firehose SSE Ingest
Date: 2026-07-08
Statut: Initialisation

## 1. Objectifs de l'audit
L'audit vise a securiser la chaine de traitement des donnees en temps reel. Les priorites sont la stabilite de l'ingestion, la validation des mecanismes de filtrage et la protection de l'infrastructure persistante contre les pics de charge imprevus.

## 2. Perimetre Technique
### Tables PostgreSQL
- firehose_events : Journalisation des evenements entrants.
- firehose_rules : Logique de filtrage et declencheurs.
- firehose_taps : Configuration des flux de sortie SSE.

### Edge Computing (Supabase Functions)
- firehose-actions : Traitement des declencheurs.
- ingest-agent, ingest-bot-hits, ingest-bot-logs : Ingestion multi-sources.
- ingest-cloudflare, ingest-vercel, ingest-wordpress : Connecteurs specifiques.
- log-connectors : Interface avec les systemes tiers.

### Frontend
- Dashboards Taps : Visualisation des flux et etat des souscriptions.

## 3. Contexte & Risques
Le systeme repose sur une ingestion continue via SSE (Server-Sent Events). Le risque principal est la saturation de la base de donnees (IOPS et stockage) lors de rafales d'evenements non filtres. Une defaillance des regles de filtrage peut entrainer un effet cascade sur les services dependants.

## 4. Methodologie
- Revue de code des fonctions Edge (Typescript).
- Audit des performances SQL (Indexation, Partitionnement potentiel).
- Analyse de la latence de bout en bout entre l'ingestion et la reception frontend.
- Test de resistance (Stress Test) sur les endpoints critiques.

## 6. Protocole de Test
1. Test de charge : Injection massive (1k req/s) pour observer la saturation DB.
2. Test d'integrite : Verification de la conformite des schemas de payload.
3. Test de latence : Mesure du delai entre l'emission d'un log et son affichage dashboard.
4. Test de rupture : Desactivation volontaire d'un edge pour tester la resilience.

## 7. Signaux a Surveiller
- Volume SQL : Nombre d'insertions par jour dans firehose_events.
- Logs Edges : Surveillance des erreurs d'execution et des timeouts.
- Taille Payload : Suivi de la taille moyenne des messages pour optimiser la bande passante.
- Taux de Rejet : Pourcentage d'evenements ecartes par firehose_rules.
