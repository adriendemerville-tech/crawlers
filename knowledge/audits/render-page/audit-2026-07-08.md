## 1. Résumé exécutif

`render-page` est fonctionnellement solide : JSON-LD complets (BlogPosting/Article/WebPage + BreadcrumbList + FAQPage auto), canonical présents sur tous les types, aucun appel LLM. Les bases SEO sont correctes.

Trois vulnérabilités bloquantes : (1) **XSS direct** via le parser Markdown maison qui injecte sans échappement les `href` et le contenu inline, (2) **SSRF** dans `bot-prerender-check` (pas de blocklist réseau), (3) **cache systématiquement bypassé** pour les landing pages. S'y ajoutent des lacunes P1 notables : pas d'ETag/304, pas d'invalidation sur publication, détection bot log-only, et `content_hash = ""` sur landing/guide.

## 2. Findings

### R1 · P0 · XSS via `markdownToHtml` — href et contenu inline non échappés
**Preuve** : `render-page/index.ts` L108
```ts
.replace(/\[([^\]]+)\]\(([^)]+)\)/g, '<a href="$2">$1</a>')
```
`$2` (URL) et `$1` (texte) sont injectés **sans** passage par `escapeHtml()`. Un article blog ou guide stockant `[clic](javascript:document.cookie)` ou `**<script>alert(1)</script>**` produit du HTML exécutable servi tel quel. Idem pour les headings L103-105.

**Impact** : XSS stocké sur toutes les routes `/blog/<slug>`, `/landing/<slug>`, `/guide/<slug>`. Tout utilisateur ayant accès en écriture à `blog_articles.content` ou `seo_page_drafts.content` peut exécuter du JS dans le HTML servi aux bots et aux partageurs sociaux.

**Fix** :
```ts
// Remplacer ligne 108 :
.replace(/\[([^\]]+)\]\(([^)]+)\)/g, (_, text, href) => {
  const safeHref = /^https?:\/\//.test(href) ? href : '#'; // URL-allowlist
  return `<a href="${escapeHtml(safeHref)}">${escapeHtml(text)}</a>`;
})
// + appliquer escapeHtml($1) sur les regexes heading/bold/italic
```
**Effort** : 0,25j · Perf : 10/10 · Sécu : **0/10** · Coût : 10/10 · Maint : 5/10

### R2 · P0 · SSRF dans `bot-prerender-check` — pas de blocklist réseau
**Preuve** : `bot-prerender-check/index.ts` L60-72
```ts
const { url } = await req.json();
let target: URL;
try { target = new URL(url); } catch { ... }
if (!['http:', 'https:'].includes(target.protocol)) { ... }
// Aucune vérification d'IP/hostname avant le fetch
const results = await Promise.all(BOTS.map((b) => probe(target.toString(), b)));
```
N'importe quelle URL HTTP(S) est acceptée : `http://localhost:5432`, `http://169.254.169.254/latest/meta-data/` (AWS IMDS), IP internes Supabase. Aucune allowlist de domaine, aucun blocage CIDR privé.

**Impact** : SSRF critique — un attaquant (ou edge function compromise) peut sonder l'infrastructure réseau interne. La fonction est appelable sans auth JWT (`Access-Control-Allow-Origin: *`).

**Fix** :
```ts
import { isIPPrivate } from "some-ip-util"; // ou impl. maison
const hostname = target.hostname;
const resolvedIp = await Deno.resolveDns(hostname, 'A').catch(() => []);
if (resolvedIp.some(isPrivate)) return jsonError("forbidden", 403);
// Et/ou allowlist: const ALLOWED_HOSTS = ['crawlers.fr', '*.crawlers.fr'];
```
**Effort** : 0,5j · Perf : 10/10 · Sécu : **0/10** · Coût : 10/10 · Maint : 7/10

### R3 · P0 · Cache `landing` jamais lu — toujours fresh
**Preuve** : `render-page/index.ts` L617-733 (bloc landing)
```ts
// Blog (L558) : if (!noCache) { const { data: cached } = ... if (cached) return cached }
// Guide (L741) : if (!noCache) { const { data: cached } = ... if (cached) return cached }
// Landing : <ABSENT> → va directement à la DB puis upsert
```
Le bloc landing écrit bien dans `prerender_cache` (L715-723) mais ne le relit jamais. Chaque requête bot sur `/landing/<slug>` déclenche un `SELECT seo_page_drafts` + `markdownToHtml` + `upsert` inutiles.

**Impact** : performance dégradée sur les landings (fresh à chaque hit), contredit l'objectif p95 < 500ms cache-hit du brief. Coût DB × fréquence bot.

**Fix** : ajouter avant L622 le même bloc cache-check présent aux lignes 558-575 (blog).

**Effort** : 0,25j · Perf : **2/10** · Sécu : 10/10 · Coût : 4/10 · Maint : 7/10

### R4 · P1 · Absence d'ETag / Cache-Control — pas de support 304
**Preuve** : `render-page/index.ts` L572-574, L612-614, L879-881, L907-909 — aucun header `ETag`, `Last-Modified`, `Cache-Control`, ni `Vary`.
```ts
headers: { ...corsHeaders, "Content-Type": "text/html; charset=utf-8", "X-Prerender": "cache-hit" }
// → pas d'ETag, pas de 304
```
**Impact** : bots et CDN re-téléchargent le HTML complet à chaque hit, même si le contenu est identique. Bande passante × fréquence de crawl. Brief §5 identifie cet écart vs Prerender.io comme un déficit avéré.

**Fix** :
```ts
const etag = `"${contentHash.slice(0, 16)}"`;
const ifNoneMatch = req.headers.get('If-None-Match');
if (ifNoneMatch === etag) return new Response(null, { status: 304, headers: { ETag: etag } });
// Ajouter sur toutes les réponses :
"ETag": etag, "Cache-Control": "public, max-age=3600, stale-while-revalidate=86400"
```
**Effort** : 0,5j · Perf : **3/10** · Sécu : 10/10 · Coût : 5/10 · Maint : 8/10

### R5 · P1 · `content_hash = ""` pour landing et guide — invalidation hachée inopérante
**Preuve** :
```ts
// landing L718 :
content_hash: "",
// guide L867 :
content_hash: "",
// blog L598-599 : correct (SHA-256 réel)
```
La colonne `content_hash` est prévue pour détecter les changements de contenu (brief §3). Vide sur deux types sur trois, elle est inutilisable pour déclencher une invalidation conditionnelle ou produire un ETag fiable.

**Fix** : calculer le SHA-256 du HTML rendu identiquement au bloc blog (L593-594) pour landing et guide.

**Effort** : 0,25j · Perf : 7/10 · Sécu : 9/10 · Coût : 8/10 · Maint : **3/10**

### R6 · P1 · Détection bot log-only — `isBot` jamais exploité
**Preuve** : `render-page/index.ts` L542-546
```ts
const isBot = BOT_UA_RE.test(userAgent);
if (isBot) {
  console.log(`[render-page] 🤖 Bot detected: ...`);
}
// isBot n'est plus référencé nulle part dans le reste du handler
```
**Impact** : impossible de prioritiser les requêtes bot (ex. skip cache-write pour les humains, ajouter `X-Robots-Tag` différencié, compter `bot_hits` dans `analytics_events`). Le brief §7 demande explicitement le hit ratio bot.

**Fix** : utiliser `isBot` pour (a) logger vers `analytics_events` (`event_type='bot_hit'`), (b) conditionner la logique de cache-write, (c) ajouter `X-Prerender-Bot: true` dans la réponse.

**Effort** : 0,5j · Perf : 8/10 · Sécu : 9/10 · Coût : 7/10 · Maint : **4/10**

### R7 · P1 · Pas d'invalidation cache sur publication d'article
**Preuve** : aucun trigger/webhook sur `blog_articles` ni `seo_page_drafts` n'appelle `render-page?route=...&nocache=true`. TTL 24h hardcodé (L603, L722, L870, L924).

**Impact** : une correction d'article publiée reste invisible pour les bots pendant 24h max. Contredit le benchmark Next.js ISR cité dans le brief (§5). En cas de coquille SEO critique, aucun moyen de régénérer sans intervention manuelle (`nocache=true`).

**Fix** : pg_net webhook ou Supabase Database Function sur `INSERT/UPDATE blog_articles` → `DELETE FROM prerender_cache WHERE route = '/blog/' || NEW.slug`. Idem `seo_page_drafts`.

**Effort** : 0,5j · Perf : 8/10 · Sécu : 10/10 · Coût : 8/10 · Maint : **4/10**

### R8 · P2 · Bloat HTML — 43 liens navigation sur toutes les pages statiques
**Preuve** : `render-page/index.ts` L403-405
```ts
<section><h2>Navigation</h2><ul>
${Object.entries(PUBLIC_ROUTES).map(([r, m]) => `<li><a href="...">${escapeHtml(m.title)}</a></li>`).join("\n")}
</ul></section>
```
Injecte ~3-4 KB de markup de navigation interne sur chaque page statique (43 routes × ~80 octets). Multiplié par le volume de requêtes bot, cela représente une bande passante inutile et un signal de maillage interne "spam" potentiel.

**Fix** : limiter à 8-10 liens contextuels (top pages + pages liées thématiquement) plutôt que toutes les routes.

**Effort** : 0,25j · Perf : 5/10 · Sécu : 10/10 · Coût : 6/10 · Maint : **5/10**

### R9 · P2 · Tarifs hardcodés dans le HTML statique — risque de désynchro
**Preuve** : `render-page/index.ts` L271-274
```ts
<tr><td>Pro Agency</td><td>29 € / mois</td>...</tr>
<tr><td>Pro Agency+</td><td>79 € / mois</td>...</tr>
```
Si les tarifs changent, la page `/tarifs` servie aux bots affichera des prix obsolètes pendant 24h (TTL cache), sans aucun mécanisme d'alerte.

**Fix** : charger les tarifs depuis une table `pricing_plans` ou invalider le cache `/tarifs` via webhook quand les tarifs changent.

**Effort** : 0,5j · Perf : 9/10 · Sécu : 10/10 · Coût : 9/10 · Maint : **3/10**

---

### R10 · P2 · Dérive ROUTE_LABELS vs PUBLIC_ROUTES — labels manquants silencieux
**Preuve** : `render-page/index.ts` L282-329 — ROUTE_LABELS contient des entrées (`/comparatif-claude-vs-crawlers`, `/cocoon`, `/comparatif-plateforme-seo-ia`, etc.) absentes de PUBLIC_ROUTES. Inversement, `/comparatif-claude-code-vs-crawlers` (L41, PUBLIC_ROUTES) n'a pas de label dans ROUTE_LABELS → fallback sur `meta.title` (L345), mais breadcrumb utilise `meta.title` complet au lieu du label court.

**Impact** : breadcrumb JSON-LD malformé (titre trop long) sur les routes sans label, breakage silencieux.

**Fix** : unifier les deux maps, ou calculer ROUTE_LABELS depuis PUBLIC_ROUTES à l'initialisation.

**Effort** : 0,25j · Perf : 9/10 · Sécu : 10/10 · Coût : 10/10 · Maint : **4/10**
