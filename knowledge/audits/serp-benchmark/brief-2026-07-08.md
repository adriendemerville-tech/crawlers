# Audit Brief — SERP Benchmark multi-providers
Date : 2026-07-08 · Auditeur : Lovable · Baseline : première itération

## 1. Objectif métier
Interroger **plusieurs providers SERP en parallèle** pour une même requête (mot-clé + géo) et fournir une **position moyenne pondérée** plus fiable qu'un provider unique — puis nourrir `keyword_universe` (position réelle vs meilleure connue) et déclencher une alerte `architect_workbench` si dérive ≥ 10 positions. C'est la source de vérité SERP pour tout ce qui suit (indexation, sea→seo, drop detector, workbench).

## 2. Cas d'usage principaux
- **Anonyme (lead magnet)** → tape une requête + son domaine, obtient la position moyenne 3 providers sans compte (pas de persistance)
- **Free / Premium** → benchmark 1-shot depuis `/console` → `IndexationMonitor` → `<SerpBenchmark>` (ref `.triggerBenchmark(domain)`)
- **Pro Agency** → benchmark récurrent via cron `refresh-serp-all` (batch 30 sites, 800ms delay, cursor pagination 240s max)
- **Système Parménion / Workbench** → consomme les `architect_workbench` créés sur dérive SERP pour prescrire des tâches (mémoire `keyword-universe-ssot-fr`)

## 3. Logique métier clé
- **4 providers** appelés en parallèle via `Promise.all` :
  - **DataForSEO** (`/v3/serp/google/organic/live/advanced`, depth=30, desktop, Basic auth)
  - **SerpApi** (`serpapi.com/search.json`, engine=google, num=30)
  - **Serper** (`google.serper.dev/search`)
  - **Bright Data** (`api.brightdata.com/request`)
- **Défaut** : `providers = ['DataForSEO', 'SerpApi', 'Serper']` — **Bright Data désactivé par défaut** (fallback opt-in)
- **Pondération** (`computeAveragedResults`) :
  - Position moyenne = `sum(positions) / count(providers qui ont trouvé l'URL)`
  - **Single-hit penalty** : si URL trouvée par 1 seul provider (sur ≥2 actifs) → `position + single_hit_penalty` (défaut 20). Justifie que sur 3 providers, une URL trouvée par 1 seul est probablement une hallucination locale (position boostée par cache/personnalisation).
  - URLs non trouvées → position 999 (bottom sort)
- **Persistance** (utilisateurs authentifiés uniquement) :
  - `serp_benchmark_results` (14 col, 3 policies) : `providers_data` jsonb full + `averaged_results` jsonb
  - `keyword_universe.current_position` mise à jour si `tracked_site_id + target_domain` fournis
  - `architect_workbench` insert automatique si `drift ≥ 10` (severity `high` ou `critical` si ≥20)
- **Accès anonyme** autorisé sur `action='benchmark'` (lead magnet) — pas d'insert DB
- **Tables** :
  - `serp_benchmark_results` (14 col, 3 policies) — historique
  - `serpapi_cache` (16 col, 4 policies) — **existe mais JAMAIS utilisée par serp-benchmark**
  - `serp_snapshots` (15 col, 2 policies) — trajectoire historique
  - `serp_geo_correlations` (29 col, 2 policies) — analyses géo
  - `serp_intent_analyses` (19 col, 5 policies) — alimentée par `analyze-serp-intents`
- **Contraintes projet** :
  - mem://features/console/serp-benchmark-fr (pondération position moyenne)
  - mem://tech/serp-benchmark-multi-providers-fr
  - mem://tech/architecture/keyword-universe-ssot-fr (SSOT keyword_universe)

## 5. Benchmark concurrentiel
- **Semrush Position Tracking** : leader — 1 provider unique (Semrush index maison), historique quotidien, alertes, mobile+desktop, tracking géo → **fort en profondeur historique mais 1 seule source**, ~130$/mois min
- **Ahrefs Rank Tracker** : idem Semrush, source unique (Ahrefs index), refresh hebdo par défaut → **pas de multi-source**
- **SE Ranking** (~55$/mois) : positionnement mid-market, 1 source, refresh quotidien
- **Nightwatch** : SERP scraping direct + rank tracking, 1 source mais transparence sur SERP features → **pas d'agrégation multi-provider**
- **AccuRanker** : ultra-rapide (refresh sur demande), 1 source
- **Aucun concurrent grand public ne propose de moyenne pondérée multi-providers avec single-hit penalty** — c'est un **différenciateur unique de Crawlers** : là où les concurrents "font confiance à leur SERP maison" (biais provider), Crawlers dit "on croise 3 sources pour minimiser le biais de personnalisation Google".

**Barre à atteindre / dépasser** : seul outil qui **croise 3-4 SERP providers en parallèle + applique une pénalité anti-hallucination** — argument marketing puissant ("position vérifiée par 3 sources indépendantes vs 1 chez Semrush/Ahrefs"). Mais **ne peut pas prétendre à ça si le cache est cassé et le tracking coût absent** — ces gaps sabotent le pitch commercial.

## Hypothèses d'audit à confirmer

1. **Cache mort** : `serpapi_cache` (16 colonnes, 4 policies) jamais utilisée par `serp-benchmark` — chaque appel refait 3 requêtes providers même sur query identique < 24h. Gaspillage direct.
2. **Zéro tracking coût** : aucun `trackPaidApiCall` dans `serp-benchmark` → coûts SERP invisibles dans `paid_api_calls` (contrairement au crawl qui track Spider/Firecrawl/Browserless proprement).
3. **Anonyme sans rate limit** : action `benchmark` autorisée sans auth (lead magnet), aucun rate limit visible → risque de drain coûts par script.
4. **Promise.all vs Promise.allSettled** : les fetchers catch leurs propres erreurs et retournent `{ error }` donc Promise.all est safe **en pratique**, mais fragile si une future modif jette. `allSettled` plus robuste.
5. **Doublon `serpapi-actions`** (217 l.) vs `fetchSerpApi` inline (35 l. dans serp-benchmark) : les 2 appellent le même endpoint SerpApi → factoriser dans `_shared/serpProviders/`.
6. **Chevauchement `fetch-serp-kpis` / `analyze-serp-intents` / `serp-benchmark`** : 3 fonctions qui touchent la SERP Google, potentielle redondance de fetch.
