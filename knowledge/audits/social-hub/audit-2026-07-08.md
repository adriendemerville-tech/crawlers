## #256 · P1 · 1j — Canva OAuth state : CSRF non vérifié côté serveur (base64 forgeable)

`canva-oauth-init` génère un `state = btoa(JSON.stringify({user_id, code_verifier, ts}))` et l'écrit **dans `canva_connections.refresh_token`** au lieu d'une table d'états dédiée. Le callback `canva-oauth-callback` décode ce base64 et vérifie uniquement le timestamp — il ne revalide jamais que le state reçu correspond à ce qui a été stocké en DB pour cet utilisateur. Un attaquant peut forger un state avec n'importe quel `user_id` et un `code_verifier` de son choix, puis déclencher un account-takeover OAuth si le `code` est intercepté.

**Fix :** Insérer l'état Canva dans `social_oauth_states` (table existante) au lieu du champ `refresh_token`, puis valider `state` par `SELECT ... WHERE state = $1 AND user_id = $2` dans le callback avant tout échange de code. Supprimer l'état après usage (même pattern que social-oauth-callback).

## #257 · P1 · 0.5j — social-oauth-callback : DELETE avant vérification d'expiry (TOCTOU)

Ligne 136 : le `DELETE` du state est exécuté **avant** le contrôle `expires_at` (ligne 139). Entre le `SELECT` (ligne 123) et le `DELETE`, deux requêtes concurrentes peuvent toutes deux retrouver le state en base et passer la vérification d'existence. La suppression du premier gagnant ne bloque pas le second si les deux écritures sont inflight simultanément — race classique sur un callback OAuth replay.

**Fix :** Remplacer `SELECT` + `DELETE` séparés par un `DELETE FROM social_oauth_states WHERE state=$1 AND platform=$2 AND expires_at > now() RETURNING *` atomique ; si aucune ligne n'est retournée, rejeter immédiatement.

## #258 · P1 · 1j — access_token et refresh_token stockés en clair (social_accounts, canva_connections)

Les colonnes `access_token` et `refresh_token` sont de type `TEXT` sans chiffrement colonne. Le `REVOKE SELECT` posé en migration 20260525 protège les rôles `authenticated`/`anon` mais le service_role (utilisé par toutes les edge functions) et tout opérateur ayant accès au dashboard Supabase lisent les tokens en clair. Une exfiltration de dump DB ou une fuite via logs expose directement des tokens permettant de publier au nom de tous les utilisateurs.

**Fix :** Chiffrer à l'écriture avec `pgcrypto.pgp_sym_encrypt(token, app_secret)` ou migrer vers Supabase Vault (`vault.create_secret`). Les edge functions déchiffrent via `pgp_sym_decrypt` ou `vault.decrypted_secrets`.

## #259 · P2 · 0.5j — Vues `social_accounts_public` et `canva_connections_public` inexistantes

`socialHub.ts` (ligne 101) interroge `social_accounts_public` et `canvaHub.ts` (ligne 58) interroge `canva_connections_public`. Aucune migration ne définit ces vues. Les requêtes échouent silencieusement masquées par le cast `as any` — le frontend reçoit `null` ou une erreur non-typée sans lever d'exception exploitable. Les vues _safe_ équivalentes (pattern `google_connections_safe`, `cms_connections_safe`) n'ont pas été créées pour le périmètre social/canva.

**Fix :** Créer les vues `social_accounts_public` et `canva_connections_public` excluant explicitement `access_token` et `refresh_token`, avec `security_invoker = true` et RLS héritée de la table sous-jacente.

## #260 · P1 · 1j — Race condition scheduler : double publication possible (absence de SELECT FOR UPDATE)

`publish-to-social` lit le post (`SELECT`), puis met à jour son statut en `publishing` (`UPDATE`) en deux opérations distinctes. Si deux invocations concurrentes reçoivent le même `post_id` (scheduler déclenché deux fois, retry applicatif), toutes deux lisent `status != 'publishing'`, passent le guard et publient en doublon sur LinkedIn/Meta. Risque de spam et de dépassement de quota API.

**Fix :** Remplacer le `SELECT` + `UPDATE` séparés par `UPDATE social_posts SET status='publishing' WHERE id=$1 AND status IN ('scheduled','draft') RETURNING *` — si aucune ligne retournée, abandonner immédiatement. Complémenter avec un index partiel sur `(id) WHERE status='scheduled'`.

## #261 · P2 · 0.5j — Aucun rate-limit sur publish-to-social (risque spam API)

La function `publish-to-social` n'appelle pas `check_rate_limit` (RPC présent et utilisé par `mcp-server`, `expert-audit`, etc.). Un utilisateur peut déclencher N publications consécutives sans délai, consommant les quotas LinkedIn/Meta jusqu'au blocage du compte connecté ou l'épuisement des credits.

**Fix :** Appeler `supabase.rpc('check_rate_limit', { p_user_id, p_action: 'social_publish', p_max_count: 10, p_window_minutes: 60 })` en tête de handler ; retourner HTTP 429 si `data.allowed === false`.

## #262 · P1 · 0.5j — SSRF : image_url non validée avant transmission aux APIs Instagram/Facebook

`publish-to-social` extrait `imageUrl = post.image_urls?.[0]` (champ libre en base) et le passe directement à `publishToInstagram` (ligne 81, champ `image_url`) et `publishToFacebook` (ligne 57, champ `url`). L'utilitaire `assertSafeUrl` existe dans `_shared/ssrf.ts` mais n'est pas importé. Un acteur malveillant ayant accès à la création de posts peut injecter une URL interne (169.254.169.254, metadata AWS, etc.) que Meta ou Instagram irait résoudre côté serveur.

**Fix :** Importer `assertSafeUrl` depuis `../_shared/ssrf.ts` et appeler `assertSafeUrl(imageUrl)` avant tout appel aux fonctions `publishTo*` ; intercepter l'exception et retourner une erreur 400.

## #263 · P2 · 0.5j — Compteur quota `social_posts_this_month` : incrément non atomique

Ligne 170 de `publish-to-social` : `social_posts_this_month: (post as any).social_posts_this_month + 1`. La valeur lue provient de l'objet `post` qui est une ligne de `social_posts` (pas de `profiles`), et l'incrément est un read-modify-write non transactionnel. Deux publications concurrentes peuvent lire la même valeur et n'incrémenter qu'une seule fois. La colonne cible (`profiles`) n'est pas du tout lue ici.

**Fix :** Remplacer par `UPDATE profiles SET social_posts_this_month = social_posts_this_month + 1 WHERE user_id = $1` — incrément SQL atomique, sans lecture préalable.

## #264 · P2 · 1j — Absence de sanitisation des captions avant publication

Les champs `content_linkedin`, `content_facebook`, `content_instagram` sont stockés en `TEXT` libre et transmis tels quels aux APIs de publication. Aucun strip des caractères de contrôle, séquences d'escape RTL, ou injection de mentions non souhaitées (`@admin`, `#banned`). Pour Instagram notamment, les captions sont rendues avec un sous-ensemble de formatage qui peut amplifier des injections de hashtags.

**Fix :** Appliquer une fonction `sanitizeCaption(text: string)` avant chaque appel API : strip des caractères de contrôle (`\x00-\x1F` sauf `\n`), limitation à la longueur max par plateforme (LinkedIn 3000, Instagram 2200, Facebook 63206), et validation regex des mentions autorisées.

## #265 · P2 · 1j — Quotas API LinkedIn/Meta non contrôlés avant publication

LinkedIn impose 100 `ugcPosts` par jour par application (quota global, pas par token). Meta limite les publications de pages selon le type de contenu et la fenêtre glissante. Aucun tracking de ces compteurs n'existe dans le codebase — ni table dédiée, ni vérification pré-publication. Un burst de publications peut déclencher un blocage temporaire de l'application entière (pas seulement du compte utilisateur).

**Fix :** Créer une table `social_api_quotas(platform, date, count)` incrémentée atomiquement à chaque publication réussie ; vérifier le count avant d'appeler l'API et retourner un statut `quota_exceeded` actionnable côté front.

## #266 · P1 · 0.5j — code_verifier PKCE stocké dans le champ `refresh_token` (confusion sémantique et risque de fuite)

`canva-oauth-init` utilise le champ `refresh_token` de `canva_connections` pour stocker temporairement le `code_verifier` PKCE (`refresh_token: codeVerifier`). Ce champ est destiné au vrai refresh token post-callback. Si le callback échoue après l'`upsert` mais avant l'`update`, le `code_verifier` persiste indéfiniment dans `refresh_token`. La migration 20260525 révoque `SELECT (refresh_token)` pour les rôles `authenticated/anon`, mais l'exposition en DB reste risquée (logs, monitoring, exports).

**Fix :** Stocker le `code_verifier` dans `social_oauth_states` avec le même TTL 10 min, colonne dédiée `pkce_verifier TEXT`. Ne jamais écrire dans `refresh_token` avant la réception du vrai token Canva.

## #267 · P2 · 0.5j — social_oauth_states : aucune policy RLS explicite (table opaque sans audit trail)

La migration 20260421 active RLS mais ne crée aucune policy : "No RLS needed — only accessed by service role". Une table RLS sans policy rejette silencieusement toutes les requêtes sauf service_role, ce qui est fonctionnellement correct, mais il n'existe aucune policy de type `DENY` explicite documentée, et aucune policy d'audit pour le rôle `postgres`. De plus, la fonction de nettoyage `cleanup_expired_oauth_states()` est `SECURITY DEFINER` sans être planifiée via `pg_cron` — les états expirés s'accumulent indéfiniment.

**Fix :** Ajouter une policy restrictive explicite `FOR ALL USING (false)` pour `authenticated` et `anon` ; planifier `SELECT public.cleanup_expired_oauth_states()` via `pg_cron` toutes les heures.

## #268 · P2 · 0.5j — Meta long-lived token : fallback silencieux sur short-lived token (durée 1-2h)

Lignes 66-72 de `social-oauth-callback` : si l'échange du short-lived vers long-lived token échoue, la fonction retourne silencieusement le short-lived (durée 1-2h) sans aucun log `console.error` ni flag en base. L'utilisateur se retrouve déconnecté de Facebook/Instagram quelques heures après l'OAuth sans explication, et les publications planifiées échouent avec un token expiré.

**Fix :** Logger explicitement l'échec du long-lived exchange (`console.error`), stocker `metadata: { token_type: 'short_lived', short_lived_reason: err }` dans `social_accounts`, et déclencher une notification front (champ `status: 'degraded'`) pour inviter l'utilisateur à reconnecter.

## #269 · P2 · 1j — canva-api-proxy : refresh token non atomique (double-refresh race)

`refreshTokenIfNeeded` lit `conn.token_expires_at`, vérifie l'expiry, appelle Canva, puis `UPDATE canva_connections`. Deux requêtes concurrentes (ex. deux onglets) peuvent toutes deux détecter le token expiré, toutes deux appeler le endpoint `/oauth/token` avec le même `refresh_token`, et l'une obtenir un nouveau token pendant que Canva invalide le `refresh_token` originel — la seconde requête obtient une erreur 400, marque la connexion `expired`, et force une re-authentification utilisateur.

**Fix :** Utiliser `UPDATE canva_connections SET status='refreshing' WHERE id=$1 AND status='active' RETURNING *` avant l'appel Canva ; si aucune ligne retournée, relire le token déjà rafraîchi par le concurrent. Remettre `status='active'` après mise à jour du token.

## #270 · P3 · 0.5j — safeServiceCall non utilisé dans publish-to-social (ownership chain incomplète)

`publish-to-social` vérifie `post.user_id = auth.userId` mais n'utilise pas `safeServiceCall` (défini dans `copilot-orchestrator/skills/registry.ts`) qui valide également la chaîne de propriété `tracked_site_id → user_id`. Un utilisateur ayant accès à un `post_id` orphelin (tracked_site supprimé, `tracked_site_id` NULL) peut le publier sans que l'appartenance au site soit vérifiée. Le pattern `safeServiceCall` impose une double validation site + user.

**Fix :** Envelopper la logique de publication dans `safeServiceCall(ctx, post.tracked_site_id, async (service, site) => {...})` ou dupliquer la vérification d'appartenance du `tracked_site_id` au `user_id` avant l'étape de publication.
